*What Alex understood:* Let me rebuild it as mechanism, not a list. An RDD is 'distributed' because property 1 splits it into partitions across machines, and 'resilient' because properties 2 and 3 together — a per-partition compute function plus a dependency list — mean Spark stored a RECIPE and a parent pointer, so a lost partition is just recomputed from its lineage instead of being lost forever. The partitioner (4) and preferred locations (5) are both optimizations: one avoids reshuffling keys you've already laid out, the other schedules compute onto the node where the data already sits so you don't ship data over the network. The reason lineage is even safe to rely on is immutability: since a parent RDD can never be mutated in place, its recipe stays valid forever, so recomputation always reproduces the same thing. Laziness is the second gear — a transformation like map doesn't run; it just adds a node to a DAG recording 'this new RDD depends on that old one via this function.' The DAG keeps growing silently until an ACTION (count, collect) fires, and THAT is what triggers execution — Spark finally reads the whole DAG and runs it. The payoff of waiting is that Spark sees the entire operation chain at once and can plan/optimize across it instead of materializing each step blindly. And the deepest 'why': Spark was born in 2009 to beat MapReduce on iterative ML, where you sweep the same data every loop. MapReduce hit disk each pass; an RDD you keep in memory and reuse skips that disk cost on every later pass — so in-memory reuse isn't a nice-to-have, it's the founding motivation and the source of the 'in-memory' name.

## Follow-up questions

**Alex:** If RDDs are immutable and every transformation makes a new RDD, and nothing runs until an action — where does the actual data physically live in between? Are the intermediate RDDs sitting in memory holding data, or is it just the recipe/DAG that exists until the action fires?

**vutr:** Grounded in the source: a transformation only records a dependency and extends the DAG — 'transformations build a DAG, actions trigger execution' — so before the action, what exists is the logical plan / recipe, not materialized results. The source reinforces this generally: 'each DataFrame object represents a logical plan to compute a dataset, but no execution occurs until the user calls a special output operation.' So intermediate RDDs are the recipe (compute function + dependencies) until an action forces computation. The source establishes that in-memory RDD reuse is foundational (the AMPLab 2009 iterative-ML motivation), but the exact rule for when a materialized RDD is kept resident in memory versus recomputed is NOT spelled out in this source — that part is unverified.

**Alex:** The source says shuffle writes to disk, not memory, 'despite Spark's in-memory reputation.' If in-memory RDD reuse is the whole founding point of Spark, how do both things coexist — in-memory reuse being foundational AND shuffles going to disk?

**vutr:** The source resolves the apparent contradiction directly. In-memory reuse is about holding a computed RDD in memory so iterative algorithms can re-read it across passes without a disk round-trip — that's the founding win. The SHUFFLE, by contrast, is a specific stage where data is repartitioned across the cluster, and there the source is explicit and quoted: 'to disk, not memory, as people often misunderstand because Spark is famous for in-memory processing.' So both are true because they describe different moments: reusing a dataset across iterations (in memory) versus exchanging data between partitions during a shuffle (to disk). Spark being 'in-memory' was always a comparative claim against MapReduce's disk-every-pass model, not a claim that nothing ever touches disk.
