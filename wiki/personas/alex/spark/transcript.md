## 2. rdd  (mastered)

(2) A compute function per partition: Spark stores the RECIPE for producing each partition, not just the data, so any partition can be recomputed on demand. (3) A list of dependencies: each RDD remembers which parent RDD(s) it came from — this is the lineage, and it's what makes the RDD 'resilient': if a partition is lost, Spark walks back up the dependency chain and rebuilds only that piece. The action is the trigger: only then does Spark look at the whole DAG it has accumulated, plan the execution, and run it. ## 4. catalyst-optimizer  (mastered)

**vutr teaches:** Catalyst is the optimizer inside Spark SQL. When you write a DataFrame or SQL query, you don't tell Spark HOW to run it, only WHAT you want. The source says a Spark DataFrame is lazy: 'Each DataFrame object represents a logical plan to compute a dataset, but no execution occurs until the user calls a special output operation.' Catalyst is the machine that turns that lazy logical plan into fast executable code, and it does it in FOUR phases (source line 5): Analysis -> Logical Optimization -> Physical Planning -> Code Generation.

PHASE 1 - ANALYSIS (resolve attributes via the Catalog). When you first write a query, Spark has an 'unresolved' logical plan. It knows you asked for a column called, say, 'revenue', but it doesn't yet know if that column really exists, what its data type is, or which table it belongs to. The Analysis phase looks everything up in the Catalog (Spark's registry of tables, columns, and their types) and 'resolves the attributes' -- meaning it binds each name you typed to a real, typed column. WHY this phase exists: you cannot optimize or plan something you haven't verified is real and correctly typed. Analysis is the sanity-check-and-label step; if you reference a column that doesn't exist, this is where it fails, before any wasted work.

PHASE 2 - LOGICAL OPTIMIZATION (predicate pushdown, projection pruning). Now that the plan is resolved, Catalyst rewrites it into an equivalent-but-cheaper logical plan. Two named tricks from the source: (a) PREDICATE PUSHDOWN -- push filters (WHERE conditions, i.e. 'predicates') as close to the data source as possible so you read and carry fewer rows. Instead of loading a whole table and THEN filtering, you filter AT read time. (b) PROJECTION PRUNING -- only read the columns you actually need ('projection' = which columns), dropping the rest early. WHY this phase exists: less data moved = less memory, less shuffle, less time. This phase is still abstract/logical -- it's rearranging WHAT to compute, not yet deciding HOW.

PHASE 3 - PHYSICAL PLANNING (cost model). One logical plan can be executed in several physically different ways -- for example, which join algorithm to use (the source elsewhere lists Sort Merge Join, Shuffle Hash Join, Broadcast Hash Join). Catalyst generates candidate physical plans and uses a COST MODEL to pick the cheapest. The cost model is why, e.g., a small table under autoBroadcastJoinThreshold (default 10MB) gets a Broadcast Hash Join instead of a shuffle. WHY this phase exists: the logical plan says WHAT; the physical plan chooses the concrete algorithm and data movement -- and the wrong algorithm on the wrong data is exactly what causes OOM or slow shuffles. The cost model is how Catalyst makes that choice with estimated numbers instead of guessing.

PHASE 4 - CODE GENERATION (Scala quasiquotes -> Java bytecode). Finally, instead of interpreting the physical plan operator-by-operator (which has per-row overhead), Catalyst GENERATES specialized Java code for the chosen plan using Scala quasiquotes, then compiles it to Java bytecode that runs on the JVM. WHY this phase exists: generated, specialized code removes interpretation overhead and lets the JVM optimize tight loops -- it's the difference between a generic interpreter and a purpose-built program for THIS query. The source notes the real-world tradeoff of this approach (via Photon's team, line 15): 'Code generation is more complicated to build and debug because the approach generates executing code at runtime; Databricks engineers need to add extra code manually to find issues.' So code gen buys speed at the cost of debuggability -- which is exactly why Photon chose interpreted C++ instead.

BONus link to AQE (line 6): Catalyst plans BEFORE running, using estimates. Adaptive Query Execution (Spark 3.0) fixes bad estimates AT runtime -- 'A shuffle/broadcast exchange creates a query stage boundary -- the pause enables re-optimization.' So AQE is like re-running Physical Planning with REAL numbers once a stage finishes, e.g. switching join strategies or coalescing shuffle partitions.

## 5. adaptive-query-execution  (mastered)

**vutr teaches:** Adaptive Query Execution (AQE) shipped in Spark 3.0 (2020). Normally the Catalyst Optimizer plans the whole query up front: Analysis resolves attributes via the Catalog, Logical Optimization does predicate pushdown and projection pruning, Physical Planning picks operators using a cost model, and Code Generation emits Java bytecode. The catch is that the cost model at planning time only has ESTIMATES of how big the data is. AQE lets Spark change the physical plan MID-FLIGHT using real numbers instead of estimates. It does three concrete things: (1) dynamically coalesces shuffle partitions, (2) switches join strategies at runtime, and (3) handles skew joins. The KEY mechanism is the query-stage boundary: a shuffle or broadcast exchange creates a boundary in the plan, and the pause at that boundary is what enables re-optimization. Why does the pause exist and why does it help? A shuffle exchange has to fully finish the map side (write its output to disk) before the reduce side can start reading. That hand-off is a natural stopping point. When Spark stops there, it can look at the shuffle output that ACTUALLY got written and measure its true size and per-partition sizes. Armed with those real runtime statistics, Spark re-optimizes the not-yet-run part of the plan. Concretely: if the real shuffle output turned out small, it can coalesce the fixed 200 default partitions (spark.sql.shuffle.partitions is 200 regardless of data size) down to fewer, right-sized partitions so it isn't launching tons of near-empty tasks. If one side of a join turns out to be under the autoBroadcastJoinThreshold (default 10MB) once measured for real, Spark can switch that join from a Sort Merge Join to a Broadcast Hash Join. And if it sees one partition is far larger than the others (skew), it can split that skewed partition apart into sub-partitions so the work spreads across tasks instead of one giant task blowing up. That last point ties directly to the OOM lesson: adding memory does not fix skew because the skewed partition still lands on one task; the right fix is to break the skewed partition apart, and AQE's skew-join handling automates exactly that at runtime.

## 9. sort-merge-join  (mastered)

**vutr teaches:** Okay, so a JOIN in Spark means matching rows from two tables that share the same key. The problem is the matching rows live on different machines, so Spark has to shuffle data around first. There are a few strategies for how to actually do the match, and the source ranks them.

Same total data, drastically fewer boundary crossings and near-zero format-conversion cost. Arrow-optimized Python UDFs (Spark 3.5) apply that same Arrow transport to make even regular-looking Python UDFs faster.
