## Comparisons
**An LLM is a probability engine, not a knowledge base — and everything downstream is a patch for that fact.** [[llm]] doesn't "know" Hanoi is Vietnam's capital; it has learned that the token is highest-probability given the preceding context, which is also why [[transformer-attention-and-parallelism|attention]] (context regardless of distance) and [[llm-training-pipeline|fine-tuning/RLHF]] (behavior shaping) exist as separate levers — one fixes what the model can attend to, the other fixes how it behaves, and neither teaches it new, fast-changing facts. [[rag]] is the third lever, and the only one aimed at freshness: it keeps facts external and retrievable instead of baked into weights, which is exactly why the source treats "RAG vs fine-tuning" as a false binary — production systems fine-tune for behavior and RAG for facts, at the same time.

**RNN's sequential bottleneck vs the Transformer's two-part fix.** The pre-2017 failure mode had one root cause — process a sentence strictly in order — with two distinct symptoms: lost long-range context (the "fox...jumped" problem) and an inability to scale training cheaply. [[transformer-attention-and-parallelism]] is explicit that these got two separate fixes bundled into one paper: attention solves the context-loss symptom by weighing relevance regardless of position, parallelism solves the scaling symptom by processing a whole sequence at once. Conflating the two into "Transformers are just faster" misses that the accuracy gain (context) and the efficiency gain (speed) are independent payoffs of the same architectural choice — and both came with a real cost the source doesn't gloss over: attention's own computational and memory overhead.

**Vector databases split on storage layout the same way OLTP and OLAP always have, but the workload doesn't sort neatly into either camp.** [[vector-database-storage-row-vs-column]] frames vector search as "retrieve related vectors, return the whole record" — an access pattern that argues for row store (Weaviate's LSM tree, ChromaDB's SQLite) the same way point-lookup OLTP does. But the same vector database's users also run ordinary analytics, which argues for column/hybrid (LanceDB's Lance format) the way OLAP does — and standard columnar formats like Parquet actively fight this workload (poor random access, unworkable row-group sizing for wide embedding columns), which is why an entirely new wave of columnar formats (Lance, Nimble, F3) exists just to fix vector storage specifically, rather than reusing Parquet's decade of tooling.

**Two ANN strategies, one underlying trade: pre-compute distance, then skip.** [[approximate-nearest-neighbor-search]] frames brute-force O(n) comparison as the honest baseline that graph-based indexing ([[hnsw]]'s sparse-to-dense multi-layer walk) and spatial partitioning (centroid-based partitions) both improve on by the same core move — precompute which vectors are close, then let a query skip the ones that clearly aren't — while accepting approximate rather than exact results. [[vector-quantization-and-compression]] is a parallel move on the storage side rather than the search side: instead of skipping comparisons, it shrinks what's stored per vector, and [[product-quantization]] is the concrete mechanism (chop into sub-vectors, replace each with its nearest codebook centroid) — both techniques trade exactness for tractability, just at different points in the pipeline (search vs storage).

**An agent's "intelligence" is really a sequence of decoupled upgrades, not one dial.** [[agent-intelligence-ladder]]'s five levels map cleanly onto separable capabilities the other agent concepts define individually: Level 1 is just [[agent-orchestration-layer|tools]] bolted onto an [[llm]]; Level 2 is context engineering inside that same orchestration layer; Level 3 is [[multi-agent-design-patterns|specialization across multiple agents]] rather than a smarter single agent; Level 4 (self-evolving) is the only level with no worked mechanism given at all in the source — it's named as "the ultimate goal," not demonstrated. The [[agentic-problem-solving-loop|5-step problem-solving loop]] (Mission → Scene → Think → Act → Observe/Iterate) is the mechanism that actually runs *inside* any level from 1 upward; the ladder describes how sophisticated the Scene-gathering and Think steps get, not a different loop at each level.

**Text-to-SQL's three challenges get one shared fix, then a further-abstracted version of that fix.** [[text-to-sql-challenges]] names natural-language ambiguity, database complexity, and the one-to-many NL-to-SQL mapping as the three failure sources — and a [[semantic-layer-and-data-modeling|semantic layer]] addresses the first two directly by pre-baking join logic and metric definitions so the model only has to reference, not infer, them. [[holistics-aql|Holistics's AQL]] pushes one step further: rather than have the model output SQL informed by the semantic layer, it has the model output a business-level query language (AQL) that the semantic layer's own well-tested converter then turns into SQL — moving the "translation-to-strict-syntax" risk entirely off the LLM and onto Holistics's deterministic conversion code. Both fixes share the same underlying dependency the source is emphatic about: neither works if the data modeling underneath the semantic layer is itself a mess — the semantic layer only formalizes decisions (which table is authoritative, how a metric is computed) that data modeling has to have already made correctly.

## Open questions
- The source explicitly declines to give a verdict on row vs column storage for vector databases, calling it a question the market hasn't settled — is there a workload-level rule (embedding dimensionality, query pattern, co-located analytics or not) that would resolve it, or will vendors keep making genuinely different bets indefinitely?
- Spatial partitioning (centroid-based) is described at a conceptual level only — the source doesn't name a specific system implementing it or compare its accuracy/speed trade-offs against [[hnsw]] directly, unlike the graph-based approach which gets a named, detailed algorithm.
- **source gap**: the source names Lance, Nimble, and F3 as new columnar formats built to outperform Parquet on vector/AI workloads but doesn't spell out the specific mechanism any of them uses to fix Parquet's random-access and row-group problems — only that they exist and share that goal.
- Level 4 of the [[agent-intelligence-ladder]] (self-evolving systems that build their own tools and agents) is named as "the ultimate goal" but the source gives no worked example or mechanism for it, unlike Levels 0–3 which each get a concrete description — is this level currently achieved by any real system, or purely aspirational?
- **source gap**: the source says Model Routing sends "complex planning" to frontier models and "simpler tasks" to cheaper ones, but doesn't spell out what signal or component actually classifies a given request as complex vs simple before routing it.
- The knowledge-graph benchmark cited for Text-to-SQL used a standardized insurance dataset — the source doesn't address whether the accuracy gain from knowledge-graph/semantic-layer grounding holds up on messier, less curated real-world schemas, which is exactly the kind of schema the source elsewhere says most companies actually have.
- **source gap**: the source states a semantic layer "automates" finding data entities and applying SQL logic but doesn't spell out the actual query-planning mechanism inside the consumption stage — how it goes from a user's selected metric/dimension to a concrete join plan.

## Synthesis
The through-line across all five posts is that every AI-adjacent system here is built by bolting a correction onto a component that is fundamentally incomplete on its own. An [[llm]] is a language probability distribution with no access to current facts, so [[rag]] and [[llm-training-pipeline|fine-tuning]] patch that from two different angles (freshness vs behavior), and an [[ai-agent]] patches it a third way by looping the model with [[agent-orchestration-layer|tools, memory, and an orchestration layer]] so it can act rather than just talk — climbing the [[agent-intelligence-ladder]] as more of that orchestration (context engineering, then specialization via [[multi-agent-design-patterns]]) gets added. A [[vector-embedding]] is how unstructured data becomes something an LLM-adjacent system can search at all, but embeddings are simultaneously too slow to search exhaustively (fixed by [[approximate-nearest-neighbor-search|ANN indexing]] like [[hnsw]]) and too large to store cheaply (fixed by [[vector-quantization-and-compression|quantization]] like [[product-quantization]]), and even the storage layout underneath them is unsettled ([[vector-database-storage-row-vs-column]], since Parquet — built for a different workload — doesn't fit). And an LLM asked to translate natural language into SQL fails for structural reasons ([[text-to-sql-challenges]]) that a [[semantic-layer-and-data-modeling|semantic layer]] fixes only if the data modeling beneath it is already sound — with [[holistics-aql|Holistics's AQL]] as the concrete case of pushing that fix one level further by giving the model a business-level language to target instead of raw SQL. None of these are one-time solved problems; each is a live trade-off the source keeps re-litigating rather than closing.
