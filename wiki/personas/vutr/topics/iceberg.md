## Comparisons
**One shared problem, three different metadata vocabularies.** Every format solves the identical translation problem — "how does a query engine treat a pile of immutable files as a table?" — with a separate, database-free metadata layer that both names the table's schema/snapshots/statistics and enforces a strict write-then-metadata, read-metadata-then-data ordering ([[open-table-formats]]). Iceberg calls its tree of these files manifest files and manifest lists rooted in a metadata JSON ([[iceberg-metadata-layer]]); Delta Lake calls it a transaction log of JSON action records ([[delta-lake]]); Hudi calls it the Timeline, an explicit three-state instant lifecycle ([[hudi-timeline]]). The vocabulary differs; the job — decouple schema and file-location knowledge from the physical bytes so schema evolution and time travel become metadata operations instead of file rewrites — does not.

**Three answers to "what is the one atomic commit step," and they trade off differently.** Object storage gives Durability free but zero cross-object atomicity, so each format has to manufacture its own single point of atomic truth ([[occ-on-object-storage]]). Iceberg pushes the requirement onto an external transactional catalog — an atomic compare-and-swap of a metadata pointer, which is why Iceberg needs Hive Metastore, Glue, Nessie, or Netflix's own Polaris ([[netflix-iceberg-ecosystem-and-migration]]) rather than raw object storage alone. Delta Lake commits by [[conditional-writes|conditional write]] directly against the object store — a put-if-absent race for the next sequential log filename, which is exactly why Delta on pre-2024 S3 needed an external DynamoDB lock service as a stand-in for a primitive S3 didn't yet have. Hudi commits by creating a suffix-less `.completed` file, but layers an *explicit* external lock (Zookeeper/DynamoDB) on top, held only for the final commit check rather than the whole transaction — the one format of the three that doesn't rely on conditional writes alone. All three land on optimistic concurrency control via a serializable-snapshot-isolation-style pattern rather than two-phase locking, because holding a lock across an object-storage round trip for the whole transaction would be untenable; Delta's own throughput ceiling ("several transactions per second," gated by put-if-absent latency) is the sharpest illustration of how expensive that round trip actually is, and Hudi's 2024 Non-Blocking Concurrency Control is a direct response to the same pressure, deferring conflict resolution to read/compaction time instead of blocking at commit.

**Copy-on-write versus merge-on-read is the same trade-off, instantiated three ways, with real production numbers behind it.** All three formats face the identical fork — rewrite the whole file (fast reads, slow writes) or write a small delta and defer reconciliation (fast writes, slower or compaction-dependent reads) — but pick different physical mechanisms: Iceberg's positional and equality delete files, Delta Lake's deletion vectors (added later, on top of an original copy-on-write-only design), and Hudi's structurally MoR-native base-file/log-file split ([[copy-on-write-vs-merge-on-read]]). Walmart's own 2023 benchmark ([[walmart-hudi-benchmark]]) is the rare case in these notes where this abstract trade-off gets measured directly in production: Hudi's compaction path beat Delta's on a heavy-upsert streaming workload, while Delta's Z-ordering (a layout optimization separate from CoW/MoR itself) let it outperform in most TPC-H queries by roughly a 40% margin — and Vu's own gloss is that Hudi has since closed much of that Z-ordering gap, so the 2023 numbers are a snapshot of a moving target, not a permanent verdict.

**Hudi's index is a genuinely different kind of pruning, not just a faster version of the other two.** Iceberg and Delta Lake both prune purely by column statistics recorded in their metadata trees (min/max, null counts) — an engine still has to know which files *might* contain a value and check. Hudi's index ([[hudi-index]]) instead maintains a direct hoodie-key-to-file-group mapping that never changes once a record's first version is written, so an update or delete is routed straight to the right file group rather than pruned toward. That is the concrete, sourced reason the notes give for why Hudi exists at all — Uber needed record-level upsert/delete speed on HDFS that stats-based pruning alone could never deliver, however good the statistics.

**"Adopting a table format" and "running the table-format spec" are not the same project, and Netflix's ecosystem is the proof.** The Iceberg specification itself is just the catalog/metadata/data hierarchy ([[iceberg-metadata-layer]]); what Netflix actually had to build around it — Polaris (a CockroachDB-backed catalog implementing Iceberg's REST spec, built specifically because Hive Metastore's pessimistic locking and Iceberg's optimistic commits fought over the same RDS instance), Janitors (TTL/snapshot/orphan cleanup), Autotune (compaction), Autolift (cross-region data movement), a signer-service access-control layer, and a five-component migration tool that moved ~1.5 million Hive tables with minimal data movement — is the operational tax of running Iceberg at hyperscale ([[netflix-iceberg-ecosystem-and-migration]]). Netflix's Write-Audit-Publish data-quality pattern ([[iceberg-branching-and-wap-implementation]]) is a further example of the same idea: it's built on Iceberg's branching mechanism specifically, not achievable with the base commit protocol alone.

**Real adoption decisions in these notes never turn on format popularity — they turn on a specific, narrow requirement.** DoorDash chose Iceberg over Delta for Flink-integration maturity, schema/partition evolution, and community — but explicitly discarded "concurrent writes" as a real differentiator once its own engineers checked and found all three formats support it ([[doordash-iceberg-realtime-migration]]). Walmart chose Hudi only after isolated, workload-specific benchmarking, and dropped Iceberg from consideration entirely over a concrete file-size cleanup problem that no amount of reputation would have surfaced ([[walmart-hudi-benchmark]]). Both are the same discipline in practice: [[evaluate-before-adopting]] against your actual workload, not the market's attention.

**A governance layer can sit above any of this, and it answers a different question than the format itself does.** Unity Catalog ([[unity-catalog]]) governs Delta Lake or Iceberg tables (plus non-table assets like ML models) from outside either format's own catalog concept — its access-control model (grant storage permissions only to UC itself, hand out short-lived credentials per request) and its own metadata store's snapshot/serializable isolation are a governance-and-sharing problem, distinct from Iceberg's own lightweight REST catalog spec (which Polaris implements) that only standardizes *how an engine finds a table's current metadata*, nothing about cross-asset governance or external sharing.

## Open questions
- Delta Lake's write-commit ceiling ("several transactions per second," gated by put-if-absent latency) is explicit in the notes. Iceberg's own equivalent ceiling is never stated — does its catalog-backed pointer swap face a comparable wall under heavy write concurrency, or does routing atomicity through a transactional database sidestep it?
- Walmart's benchmark is explicitly a 2023 snapshot that Vu says is likely stale (Hudi has since added Z-ordering and better file-group metadata) — by how much has the gap actually closed, and does Hudi's win on Walmart's specific workloads still hold?
- If [[hudi-index]] is the feature that most sets Hudi apart, why hasn't it translated into the wide adoption Iceberg and Delta Lake enjoy? The notes note the gap without resolving it.
- **source gap** ([[hudi-index]]): the source names that Hudi supports multiple index types (via a link to Hudi's own docs) but never itself describes how any specific index type maps hoodie keys to file groups internally.
- **source gap** ([[iceberg-branching-and-wap-implementation]]): the source describes table-level and catalog-level (Nessie) branching as two separate options but doesn't say what happens, or which takes precedence, if both are configured on the same table at once.
- **source gap** ([[netflix-iceberg-ecosystem-and-migration]]): the source says Autotune "understands how files should be compacted" from its compaction config but doesn't spell out the actual decision rule for when a table crosses the threshold that triggers compaction versus just reacting to any write event.
- **source gap** ([[conditional-writes]] / [[delta-lake]]): the source names the pre-August-2024 external DynamoDB lock service Delta Lake on S3 relied on but doesn't detail the actual locking protocol it implemented to guarantee single-writer semantics for a given log ID.
- **source gap** ([[unity-catalog]]): the source names write-through caching and database sharding for mutable metadata but doesn't specify the actual shard key, shard count, or how a client is routed to the right shard.
- Does Netflix's Write-Audit-Publish pattern, which the notes tie specifically to Iceberg's hidden-snapshot/branching mechanics, have any real equivalent for teams standardized on Delta Lake or Hudi instead — or does that data-quality pattern genuinely require Iceberg specifically, as the notes imply but never test against the other two formats directly?

## Synthesis
Iceberg, Delta Lake, and Hudi are three answers to one shared bet: put a separate, database-free metadata layer on top of object storage ([[open-table-formats]]) and make that layer — not the object store — responsible for Atomicity and Isolation, since Durability and (mostly) Consistency arrive for free. Where they diverge is the concrete mechanism for that one atomic commit ([[occ-on-object-storage]]): Iceberg's catalog-backed pointer swap ([[iceberg-metadata-layer]]), Delta Lake's conditional-write log entry ([[delta-lake]], [[conditional-writes]]), and Hudi's lock-guarded completed-file ([[hudi-timeline]]) — and the concrete mechanism for handling row-level change, split the identical CoW/MoR way three different physical ways ([[copy-on-write-vs-merge-on-read]]), with Walmart's own numbers ([[walmart-hudi-benchmark]]) showing that trade-off actually cashing out in production rather than staying theoretical. Hudi's index ([[hudi-index]]) is the one feature in this set that isn't just a variation on stats-based pruning — it exists because Uber's original problem (fast record-level upserts on HDFS) was never really the same problem Netflix built Iceberg to solve (query correctness and atomic transactions at Hive's breaking point) or Databricks built Delta Lake to solve (a transactional storage layer under Spark). Running any of these formats well at scale is its own project on top of the spec — Netflix's Polaris/Janitors/Autotune/Autolift/migration-tooling ecosystem ([[netflix-iceberg-ecosystem-and-migration]]) and its branching-based WAP pattern ([[iceberg-branching-and-wap-implementation]]) are what that actually costs — and a governance layer like Unity Catalog ([[unity-catalog]]) can sit above any of the three formats to answer a still further question (who can see and share what) that none of the formats' own catalogs were built to answer. The one instrument every real adoption decision in these notes actually used, DoorDash's and Walmart's alike, was the same: [[evaluate-before-adopting]] against the organization's specific workload, never the format with the most attention.
