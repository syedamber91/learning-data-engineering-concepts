## Comparisons
**Sort-merge join vs. shuffle hash join vs. broadcast join vs. bucket join.** These four aren't peers on one menu -- they're layered corrections to the same underlying trade-off. [[sort-merge-join-vs-shuffle-hash-join|Sort-merge join]] is Spark's default because it can spill a too-big partition to disk instead of dying, while shuffle hash join builds an in-memory hash table per partition and throws OOM the moment a partition -- skewed or not -- won't fit; that's exactly why SHJ was pulled in Spark 1.6 and only came back in 2.0 as an opt-in, gated by `spark.sql.adaptive.maxShuffledHashJoinLocalMapThreshold` defaulting to 0. [[broadcast-join-and-bucket-join|Broadcast join]] sidesteps the shuffle both of those strategies pay for -- if one side is small enough (`spark.sql.autoBroadcastJoinThreshold`, 10MB default), Spark ships the whole table to every executor instead of repartitioning both sides. Bucket join doesn't eliminate the shuffle either, it just moves it earlier: pre-bucketing at write time via `bucketBy()` plus `saveAsTable()` means the join-time `Exchange` step never has to run, at the cost of a slower write. [[adaptive-query-execution|AQE]] is the thread that ties all four together -- it can watch real post-shuffle sizes and convert a planned sort-merge join into shuffle hash join or broadcast join mid-query, something the static, pre-run Catalyst plan could never safely do.

**Static vs. dynamic resource allocation.** [[resource-allocation-and-scheduling-modes|Static allocation]] reserves a fixed executor count for the SparkContext's entire life -- simple, and safe, because an executor only ever exits when the whole application is done, so nothing it was holding can be stranded. Dynamic allocation trades that safety for elasticity: it grows executor count exponentially (1, 2, 4, 8...) against a pending-task backlog and shrinks it after `spark.dynamicAllocation.executorIdleTime`, but removing an executor mid-run means anything it held -- shuffle output, cached partitions -- is at risk of forcing a recompute. That risk is exactly why the [[shuffle-writes-to-disk-and-external-shuffle-service|external shuffle service]] exists: it decouples shuffle-file serving from executor lifetime so dynamic allocation can reclaim executors aggressively without losing shuffle output, while cached data gets no equivalent protection -- only a manual "never remove this executor" opt-out. Static allocation never needed this machinery because it never creates the mid-run removal problem in the first place.

**PySpark's Py4j overhead vs. Spark Connect's gRPC approach.** [[pyspark-architecture-and-py4j|PySpark]] is a two-process wrapper: a Python process and a JVM process talk over Py4j (ports 25334/25333), and Py4j's JVM *is* the actual Spark driver -- meaning the client machine has to carry the full jar/JRE dependency tree and keep that driver alive for the run's entire lifetime. [[spark-connect-architecture|Spark Connect]] answers the same problem differently: instead of spawning a local JVM driver, the client serializes an unresolved logical plan to protobuf and ships it over gRPC to a long-running remote server that holds the real driver and executors, streaming results back as Arrow batches. The trade Connect makes is real, not free -- it drops RDD/SparkContext access and fixes compute allocation at server-start time, so every client sharing that server competes for one pool, versus Py4j's per-application isolation. Vu frames Connect as structurally similar to talking to a database over JDBC rather than "PySpark but faster," and it's the same lineage that produced Arrow-optimized and Pandas UDFs -- all attempts to shrink the IPC tax Py4j imposes rather than eliminate the two-process model outright, until Connect eliminates it by relocating the driver entirely.

**Catalyst's code generation vs. Photon's vectorized execution.** [[catalyst-optimizer-phases|Catalyst]]'s final phase compiles the chosen physical plan straight to JVM bytecode -- fast at runtime, but the generated code fuses operators into fused, row-at-a-time loops that are hard to debug and impossible to introspect mid-execution. [[photon-vectorized-engine|Photon]] deliberately rejected that path: Databricks prototyped both and the vectorized approach took weeks versus two months for code generation, precisely because runtime-generated code needed extra instrumentation just to debug. Photon's payoff is that vectorization keeps operator boundaries intact and lets it inspect and react to each column batch's actual runtime shape -- a NULL-free batch, an all-ASCII string batch -- something a pre-compiled bytecode plan can't do. The two aren't really competitors: Photon slots in as a set of native operators Catalyst's plan can call, falling back to legacy SparkSQL code generation wherever Photon doesn't have an equivalent operator.

**AQE's query-level adaptivity vs. Photon's batch-level adaptivity.** Both mechanisms exist because a static plan built from stale or missing statistics is a liability, but they intervene at different granularities. [[adaptive-query-execution|AQE]] can only re-plan at a [[jobs-stages-tasks-dag-and-dependencies|stage boundary]] -- the shuffle materialization pause is the one checkpoint where real numbers exist, so AQE coalesces partitions, splits skewed ones, or swaps join strategies once per boundary, not continuously. [[photon-vectorized-engine|Photon]]'s adaptivity operates one level down and far more often: every column batch it processes gets inspected on the spot, dropping branches for NULL-free data or skipping position-list lookups when nothing was filtered out. AQE reacts to a whole stage's aggregate statistics; Photon reacts to what's actually sitting in the batch in front of it right now.

**Native local-disk shuffle vs. Uber's Remote Shuffle Service.** [[shuffle-writes-to-disk-and-external-shuffle-service|Native Spark shuffle]] and [[uber-remote-shuffle-service|RSS]] both exist to answer "where does map output live," but they diverge on which side does the fan-out. Stock Spark writes each mapper's output to its own local disk via the [[spark-shuffle-write-read-and-tuning|sort shuffle manager's]] single-file-plus-index-file layout, and a reducer fans *out* to every mapper host holding a slice of its partition — the external shuffle service only decouples that serving from executor lifetime, it doesn't change the many-mappers-to-one-reducer shape. RSS inverts the fan-out instead: every mapper writing data for a given partition sends it to the *same* single RSS server, so a reducer talks to exactly one remote host per partition. The two aren't really competing implementations of the same idea — RSS is what you build once local disk itself (not just executor lifetime) becomes the bottleneck, at Uber's ~10PB/day scale where a single machine's 1TB SSD is a hard ceiling native shuffle can't get past.

**AQE's dynamic coalescing as production infrastructure vs. as a query-tuning feature.** Most of this topic's AQE material treats coalescing/splitting/switching as things a single query's plan does at a stage boundary. [[airbnb-aqe-shuffle-partition-case-study|Airbnb's ingestion migration]] is the same mechanism applied one level up: a *framework* serving 1,000+ tables at wildly different scales can't have one correct static `spark.sql.shuffle.partitions` value at all, so AQE's coalescing isn't optimizing an already-reasonable plan, it's substituting for a per-table human tuning decision that doesn't scale to a shared framework.

**Micro-batch triggers vs. AQE's stage-boundary re-planning — two different senses of "how often does Spark re-evaluate."** [[spark-structured-streaming-microbatch-and-triggers|A streaming trigger]] controls how often Spark starts an entirely new job — a fresh logical/physical plan built from scratch against whatever's newly arrived — while [[adaptive-query-execution|AQE]] re-plans *within* a single job, at the pause a shuffle boundary already forces. They compose rather than compete: each micro-batch triggered by Structured Streaming is itself a small batch job that goes through the ordinary Catalyst-then-AQE pipeline, so a badly-chosen trigger interval (too many tiny batches) multiplies AQE's own per-stage replanning overhead by however many extra batches it created.

## Synthesis
Spark's whole architecture traces back to a specific 2009 diagnosis at UC Berkeley's AMPLab: [[spark-origin-and-mapreduce-limitations|MapReduce's]] disk-based hand-off between Map and Reduce was a durability choice that made iterative ML and interactive queries unbearably slow, so the fix paired a functional API with an in-memory sharing engine built around [[rdd-fundamentals-and-properties|RDDs]] -- immutable, partitioned, lineage-tracked collections whose five properties (partitions, computation function, dependencies, partitioner, preferred locations) replace disk replication with recomputation as the fault-tolerance mechanism. That immutability is inseparable from [[lazy-evaluation-transformations-actions|laziness]]: nothing computes until an action fires, which is what lets Spark see the whole transformation chain and build a proper DAG rather than executing step by step the way MapReduce forced. That DAG resolves into the strict [[jobs-stages-tasks-dag-and-dependencies|job -> stage -> task]] hierarchy, cut precisely at wide-dependency shuffle boundaries, scheduled by a [[spark-application-architecture-and-execution-modes|driver/executor]] split whose physical placement (cluster, client, local, or the relocated-driver [[spark-connect-architecture|Spark Connect]] model) is a separate question from the DAG itself. Once a job is defined, [[catalyst-optimizer-phases|Catalyst]] resolves, rule-optimizes, cost-optimizes, and code-generates it into a physical plan, and because table statistics are frequently stale, [[adaptive-query-execution|AQE]] uses the forced pause at every stage boundary to re-plan with real numbers -- coalescing or splitting partitions, and switching between [[sort-merge-join-vs-shuffle-hash-join|sort-merge join, shuffle hash join]], and [[broadcast-join-and-bucket-join|broadcast or bucket join]] once actual sizes are known; [[airbnb-aqe-shuffle-partition-case-study|Airbnb's own ingestion migration]] shows that same coalescing mechanism substituting for per-table manual tuning across a 1,000+ table framework, not just fixing one query's plan. Execution itself runs inside an [[executor-memory-model-and-caching|executor memory model]] where usable memory is a fraction of a fraction of what you configured, made worse by JVM object overhead that [[tungsten-and-jvm-object-overhead|Tungsten's]] off-heap UnsafeRow layout exists to avoid -- and that same memory ceiling is what turns [[data-skew-and-oom|skew into OOM]] specifically for hash-based joins, a failure mode [[spark-scheduler-oom-nondeterminism-and-skew-diagnostics|whose apparent randomness]] actually traces to the scheduler filling task slots with no partition-size awareness, so the same skewed job can pass or fail purely on how its task count divides against cluster slot count, and be diagnosed after the fact via the Spark UI's stage-level Input Size/Shuffle Read/Spill metrics. The shuffle that produces skewed partitions in the first place has its own internal write/read mechanism -- [[spark-shuffle-write-read-and-tuning|hash vs. sort shuffle managers]] deciding between many small per-reducer files and one sorted file plus an index -- and [[shuffle-writes-to-disk-and-external-shuffle-service|shuffle output]] gets decoupled from executor lifetime via the external shuffle service so that [[resource-allocation-and-scheduling-modes|dynamic resource allocation]] can reclaim idle executors without forcing recomputation; [[uber-remote-shuffle-service|Uber's Remote Shuffle Service]] takes that same decoupling further, moving shuffle off local disk entirely once disk itself becomes the bottleneck at petabyte scale, all under scheduling and [[data-locality-and-speculative-execution|locality/speculation]] policies that decide which executor a task lands on and what happens when one runs too slow. On the frontier, [[photon-vectorized-engine|Photon]] rejects Catalyst's code-generation path in favor of C++ vectorized batch execution with its own batch-level adaptivity, while on the client side [[pyspark-architecture-and-py4j|PySpark's]] two-process Py4j model and the [[python-udf-overhead-and-arrow-optimization|serialization tax it imposes on UDFs]] are being progressively dissolved by Arrow-optimized and Pandas UDFs and, more structurally, by Spark Connect's gRPC-and-protobuf redesign -- two different answers, at two different layers of the stack, to the same underlying question of how to keep Spark fast once you stop trusting a plan made before execution actually started. Structured Streaming is the same batch machinery run on a timer: each [[spark-structured-streaming-microbatch-and-triggers|trigger]] packages newly-arrived data as an ordinary, small batch DataFrame that goes through the same Catalyst/AQE pipeline, with [[spark-watermark-event-time-and-windowing|watermarks]] supplying the bounded search scope [[spark-stateful-operations-and-state-store|stateful operations]] (aggregation, stream-stream joins, dedup) need to know when their state can safely be cleaned up, backed by either the default JVM-heap-bound state store or RocksDB since 3.2 -- and the whole thing survives failure via [[spark-checkpointing-and-exactly-once|an offsets/commits checkpoint log]] whose exactly-once guarantee, Vu is explicit, depends on the source and sink holding up their own end, not on Spark alone.
