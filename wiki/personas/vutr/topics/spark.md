## Comparisons
**Sort-merge join vs. shuffle hash join vs. broadcast join vs. bucket join.** These four aren't peers on one menu -- they're layered corrections to the same underlying trade-off. [[sort-merge-join-vs-shuffle-hash-join|Sort-merge join]] is Spark's default because it can spill a too-big partition to disk instead of dying, while shuffle hash join builds an in-memory hash table per partition and throws OOM the moment a partition -- skewed or not -- won't fit; that's exactly why SHJ was pulled in Spark 1.6 and only came back in 2.0 as an opt-in, gated by `spark.sql.adaptive.maxShuffledHashJoinLocalMapThreshold` defaulting to 0. [[broadcast-join-and-bucket-join|Broadcast join]] sidesteps the shuffle both of those strategies pay for -- if one side is small enough (`spark.sql.autoBroadcastJoinThreshold`, 10MB default), Spark ships the whole table to every executor instead of repartitioning both sides. Bucket join doesn't eliminate the shuffle either, it just moves it earlier: pre-bucketing at write time via `bucketBy()` plus `saveAsTable()` means the join-time `Exchange` step never has to run, at the cost of a slower write. [[adaptive-query-execution|AQE]] is the thread that ties all four together -- it can watch real post-shuffle sizes and convert a planned sort-merge join into shuffle hash join or broadcast join mid-query, something the static, pre-run Catalyst plan could never safely do.

**Static vs. dynamic resource allocation.** [[resource-allocation-and-scheduling-modes|Static allocation]] reserves a fixed executor count for the SparkContext's entire life -- simple, and safe, because an executor only ever exits when the whole application is done, so nothing it was holding can be stranded. Dynamic allocation trades that safety for elasticity: it grows executor count exponentially (1, 2, 4, 8...) against a pending-task backlog and shrinks it after `spark.dynamicAllocation.executorIdleTime`, but removing an executor mid-run means anything it held -- shuffle output, cached partitions -- is at risk of forcing a recompute. That risk is exactly why the [[shuffle-writes-to-disk-and-external-shuffle-service|external shuffle service]] exists: it decouples shuffle-file serving from executor lifetime so dynamic allocation can reclaim executors aggressively without losing shuffle output, while cached data gets no equivalent protection -- only a manual "never remove this executor" opt-out. Static allocation never needed this machinery because it never creates the mid-run removal problem in the first place.

**PySpark's Py4j overhead vs. Spark Connect's gRPC approach.** [[pyspark-architecture-and-py4j|PySpark]] is a two-process wrapper: a Python process and a JVM process talk over Py4j (ports 25334/25333), and Py4j's JVM *is* the actual Spark driver -- meaning the client machine has to carry the full jar/JRE dependency tree and keep that driver alive for the run's entire lifetime. [[spark-connect-architecture|Spark Connect]] answers the same problem differently: instead of spawning a local JVM driver, the client serializes an unresolved logical plan to protobuf and ships it over gRPC to a long-running remote server that holds the real driver and executors, streaming results back as Arrow batches. The trade Connect makes is real, not free -- it drops RDD/SparkContext access and fixes compute allocation at server-start time, so every client sharing that server competes for one pool, versus Py4j's per-application isolation. Vu frames Connect as structurally similar to talking to a database over JDBC rather than "PySpark but faster," and it's the same lineage that produced Arrow-optimized and Pandas UDFs -- all attempts to shrink the IPC tax Py4j imposes rather than eliminate the two-process model outright, until Connect eliminates it by relocating the driver entirely.

**Catalyst's code generation vs. Photon's vectorized execution.** [[catalyst-optimizer-phases|Catalyst]]'s final phase compiles the chosen physical plan straight to JVM bytecode -- fast at runtime, but the generated code fuses operators into fused, row-at-a-time loops that are hard to debug and impossible to introspect mid-execution. [[photon-vectorized-engine|Photon]] deliberately rejected that path: Databricks prototyped both and the vectorized approach took weeks versus two months for code generation, precisely because runtime-generated code needed extra instrumentation just to debug. Photon's payoff is that vectorization keeps operator boundaries intact and lets it inspect and react to each column batch's actual runtime shape -- a NULL-free batch, an all-ASCII string batch -- something a pre-compiled bytecode plan can't do. The two aren't really competitors: Photon slots in as a set of native operators Catalyst's plan can call, falling back to legacy SparkSQL code generation wherever Photon doesn't have an equivalent operator.

**AQE's query-level adaptivity vs. Photon's batch-level adaptivity.** Both mechanisms exist because a static plan built from stale or missing statistics is a liability, but they intervene at different granularities. [[adaptive-query-execution|AQE]] can only re-plan at a [[jobs-stages-tasks-dag-and-dependencies|stage boundary]] -- the shuffle materialization pause is the one checkpoint where real numbers exist, so AQE coalesces partitions, splits skewed ones, or swaps join strategies once per boundary, not continuously. [[photon-vectorized-engine|Photon]]'s adaptivity operates one level down and far more often: every column batch it processes gets inspected on the spot, dropping branches for NULL-free data or skipping position-list lookups when nothing was filtered out. AQE reacts to a whole stage's aggregate statistics; Photon reacts to what's actually sitting in the batch in front of it right now.

## Open questions
- Vu raised executor memory from 2GB to 4GB in his TPC-H aggregation case study and spill stayed exactly the same -- he flags this as counterintuitive and unresolved, guessing it's tied to the [[sort-merge-join-vs-shuffle-hash-join|SortAggregate]] mechanism rather than raw memory pressure, but the notes never confirm the actual cause (see [[data-skew-and-oom]], [[executor-memory-model-and-caching]]).
- [[adaptive-query-execution|AQE]] is documented as splitting skewed partitions at a stage boundary, but the notes are explicit that the actual skew-join-split algorithm isn't spelled out beyond that one line -- what triggers the split, and by what threshold, is left open.
- [[executor-memory-model-and-caching|Cached data]] surviving executor removal is described as a "stated direction, not a shipped mechanism" -- off-heap, lifetime-independent caching is where the source says things are headed, not where they currently are. When or whether that lands is unresolved.
- [[data-locality-and-speculative-execution|Delay scheduling's]] wait window before falling back to a worse locality level is described only as "short and configurable" -- no actual duration or default is given, leaving the real cost/benefit tuning point unspecified.
- Vu's own verdict on [[spark-connect-architecture|Spark Connect]] stops short of a strong claim -- he says it "won't replace the last decade of Spark application development outright" but opens new use cases (IoT, non-JVM/non-Python clients); whether it becomes the default deployment model or stays a niche path is left genuinely open.
- **source gap** ([[adaptive-query-execution]]): Names 'splitting oversized partitions' as one of AQE's three runtime actions but doesn't spell out the detection/threshold mechanism for how AQE decides a partition is oversized and how it splits it — only calls it 'the mirror case' of coalescing.
- **source gap** ([[broadcast-join-and-bucket-join]]): Doesn't spell out how Spark/bucketBy() chooses the hash function or how the number of buckets affects skew/performance — only that bucketing 'hashes a column's values into a fixed number of buckets.'
- **source gap** ([[catalyst-optimizer-phases]]): Phase 4 (code generation) is named but the actual mechanism — how the physical plan/expression tree becomes Java bytecode (e.g. whole-stage codegen) — isn't spelled out beyond 'generates actual Java bytecode for execution.'
- **source gap** ([[data-locality-and-speculative-execution]]): Delay scheduling is explained conceptually (wait, then relax to a farther locality level) but the note itself flags that the source never gives the actual wait duration or how it's computed per locality level — only 'short and configurable.'
- **source gap** ([[data-skew-and-oom]]): AQE's skew-partition-splitting algorithm is referenced but not explained beyond one line ('splits huge partitions into smaller ones'); the note explicitly says the sources don't spell out the mechanism.
- **source gap** ([[data-skew-and-oom]]): The result that raising executor memory 2GB→4GB cut task time but did not reduce spill is left causally unresolved by the source itself (an unconfirmed guess tied to SortAggregate) — not reconstructable as a mechanism from this note.
- **source gap** ([[executor-memory-model-and-caching]]): The finding that doubling executor memory (2GB→4GB, same cores) didn't shrink spill at all is flagged by the source as counterintuitive/unresolved — the underlying causal mechanism isn't actually explained, only guessed at.
- **source gap** ([[photon-vectorized-engine]]): Lists 'shuffle encoding adapts to observed data patterns' as one of Photon's batch-level adaptivity behaviors but doesn't explain what triggers the adaptation or what the alternative encodings are.
- **source gap** ([[shuffle-writes-to-disk-and-external-shuffle-service]]): Explains why shuffle map output is written to local disk and how the external shuffle service decouples serving from executor lifetime, but doesn't spell out the disk-write mechanics themselves (partition/index file layout, sort-based shuffle write path) — only that 'an executor writes its map outputs to local disk.'

## Synthesis
Spark's whole architecture traces back to a specific 2009 diagnosis at UC Berkeley's AMPLab: [[spark-origin-and-mapreduce-limitations|MapReduce's]] disk-based hand-off between Map and Reduce was a durability choice that made iterative ML and interactive queries unbearably slow, so the fix paired a functional API with an in-memory sharing engine built around [[rdd-fundamentals-and-properties|RDDs]] -- immutable, partitioned, lineage-tracked collections whose five properties (partitions, computation function, dependencies, partitioner, preferred locations) replace disk replication with recomputation as the fault-tolerance mechanism. That immutability is inseparable from [[lazy-evaluation-transformations-actions|laziness]]: nothing computes until an action fires, which is what lets Spark see the whole transformation chain and build a proper DAG rather than executing step by step the way MapReduce forced. That DAG resolves into the strict [[jobs-stages-tasks-dag-and-dependencies|job -> stage -> task]] hierarchy, cut precisely at wide-dependency shuffle boundaries, scheduled by a [[spark-application-architecture-and-execution-modes|driver/executor]] split whose physical placement (cluster, client, local, or the relocated-driver [[spark-connect-architecture|Spark Connect]] model) is a separate question from the DAG itself. Once a job is defined, [[catalyst-optimizer-phases|Catalyst]] resolves, rule-optimizes, cost-optimizes, and code-generates it into a physical plan, and because table statistics are frequently stale, [[adaptive-query-execution|AQE]] uses the forced pause at every stage boundary to re-plan with real numbers -- coalescing or splitting partitions, and switching between [[sort-merge-join-vs-shuffle-hash-join|sort-merge join, shuffle hash join]], and [[broadcast-join-and-bucket-join|broadcast or bucket join]] once actual sizes are known. Execution itself runs inside an [[executor-memory-model-and-caching|executor memory model]] where usable memory is a fraction of a fraction of what you configured, made worse by JVM object overhead that [[tungsten-and-jvm-object-overhead|Tungsten's]] off-heap UnsafeRow layout exists to avoid -- and that same memory ceiling is what turns [[data-skew-and-oom|skew into OOM]] specifically for hash-based joins, while [[shuffle-writes-to-disk-and-external-shuffle-service|shuffle output]] gets decoupled from executor lifetime via the external shuffle service so that [[resource-allocation-and-scheduling-modes|dynamic resource allocation]] can reclaim idle executors without forcing recomputation, all under scheduling and [[data-locality-and-speculative-execution|locality/speculation]] policies that decide which executor a task lands on and what happens when one runs too slow. On the frontier, [[photon-vectorized-engine|Photon]] rejects Catalyst's code-generation path in favor of C++ vectorized batch execution with its own batch-level adaptivity, while on the client side [[pyspark-architecture-and-py4j|PySpark's]] two-process Py4j model and the [[python-udf-overhead-and-arrow-optimization|serialization tax it imposes on UDFs]] are being progressively dissolved by Arrow-optimized and Pandas UDFs and, more structurally, by Spark Connect's gRPC-and-protobuf redesign -- two different answers, at two different layers of the stack, to the same underlying question of how to keep Spark fast once you stop trusting a plan made before execution actually started.
