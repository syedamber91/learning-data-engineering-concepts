## Comparisons
**Same starting premise, opposite scaling answer.** [[gfs]] and [[hdfs]] are close cousins — both assume [[component-failure-as-normal|failure is normal]], both centralize metadata in one RAM-resident coordinator (GFS's master, HDFS's NameNode), both default to 3-way replication, both fix a large chunk/block size (64MB vs 128MB) specifically to keep master-contact frequency and metadata size down, and both deliberately decline to persist chunk/block *locations* on the coordinator, polling them from chunkservers/DataNodes instead ([[gfs-metadata-and-operation-log]], [[hdfs-datanode-handshake-and-heartbeat]]). [[amazon-s3]] answers the same durability-at-scale problem completely differently: it has no single coordinator at all. It's 350+ microservices partitioning a flat keyspace by lexicographic prefix ([[prefix-as-folders]]), which is precisely what sidesteps the RAM ceiling that caps HDFS in practice — [[hdfs-namenode-scaling-limit]] (grounded in a sibling raw group's Uber case study, not this group's posts) puts that ceiling at roughly 10PB, worsening past 50-100PB. GFS and HDFS's coordinator-bound design was a rational trade for the 2000s data-center world these papers describe; S3's disaggregation is the trade cloud vendors could only make once they were operating at multi-tenant, multi-region scale.

**Durability by replication vs durability by erasure coding.** GFS and HDFS both buy durability the same way: whole-copy replication (3 full copies by default), reconstructed from a surviving replica when one fails ([[gfs-replica-management-and-fault-tolerance]], [[hdfs-block-placement-and-replication-management]]). S3 buys the *same* property — surviving the loss of some fragments — for a fraction of the storage overhead, via [[object-storage-durability-erasure-coding|erasure coding]]: split data into X primary fragments, generate Y parity fragments mathematically, and tolerate losing up to Y of them. Vu's own numbers make the gap concrete: 3-way replication costs 300% storage overhead; a typical (X=4, Y=2) erasure-coding scheme costs only 150%. GFS and HDFS could not easily adopt erasure coding themselves without abandoning the assumption that drives their whole design — sequential appends to open files, not the closed, versioned objects erasure coding fragments cleanly.

**Consistency: mutation ordering vs a staleness-checking cache.** GFS's [[gfs-consistency-model|consistency model]] is earned through *coordination*: a lease-holding primary imposes one mutation order that every replica must apply identically ([[gfs-lease-and-write-pipeline]]), plus chunk version numbers to exclude replicas that missed writes. S3's original eventual-consistency window came from an entirely different source — a metadata cache that could serve stale reads after a PUT/DELETE — and [[s3-strong-consistency|closing it]] took adding a staleness-check component in front of that cache, not adopting anything like GFS's lease/ordering machinery. HDFS sits closer to GFS here: its own [[hdfs-client-read-write-and-lease|write lease]] (with soft/hard timeout limits) plays a comparable role to GFS's chunk lease, just scoped to whole-file writes rather than per-chunk mutation ordering.

**Metadata durability: operation log vs image/journal/checkpoint — nearly identical vocabulary, different auxiliary roles.** GFS persists namespace mutations to an [[gfs-metadata-and-operation-log|operation log]], checkpointed periodically into a B-tree-like structure, replicated to remote machines, with checkpoint creation happening in a background thread so it never blocks live mutations. HDFS's [[hdfs-image-and-journal|image/journal/checkpoint]] model is explicitly the same idea under different names — Vu draws the parallel directly in the HDFS post. Where HDFS goes further is formalizing *who* does the periodic journal-compaction work as dedicated roles: a [[hdfs-checkpointnode-and-backupnode|CheckpointNode]] (pulls image+journal, merges, pushes the result back) or a BackupNode (stays synced live via a streamed journal, doubling as a read-only NameNode). GFS's paper, at least as Vu covers it, doesn't name an equivalent standing role — checkpointing is just something the master itself does in a background thread.

**Atomicity: none of these three offers multi-object transactions, and each pushes the problem somewhere different.** GFS gets atomicity per mutation via its lease-ordered write pipeline and, for concurrent appends specifically, via [[gfs-record-append|record append's]] at-least-once guarantee (accepting duplicates as the price of lock-free concurrent appends). S3 guarantees atomicity only per single object — a PUT either fully succeeds or leaves nothing — and hands the multi-object problem to [[s3-atomicity-and-conditional-writes|conditional writes]], which is exactly the primitive Delta Lake and Apache Hudi lean on to create atomic metadata objects and simulate table-level transactions on top of a store that has none natively. HDFS's answer is closer to GFS's: single-writer-per-file plus an append-only close-then-immutable model, no analogue to S3's conditional write because HDFS was never trying to support many independent writers racing for the same key.

## Open questions
- The GFS paper (2003) predates S3, HDFS-at-scale, and the whole erasure-coding-over-replication argument the S3 post makes — Vu's own note flags that some details "may be changed or updated now," but the post doesn't say which GFS mechanisms Google has since replaced internally (its Colossus successor is covered separately — see [[google-infrastructure]] — but this group's posts don't connect the dots explicitly).
- **source gap**: the GFS post describes chunk version numbers detecting stale replicas but doesn't explain what happens if the master itself crashes between granting a new lease (and bumping the version) and that bump reaching all replicas — is there a window where the master's own record of the current version could itself be stale on restart?
- **source gap** ([[gfs-snapshot-and-namespace-locking]]): the source explains snapshot's copy-on-write lazily triggers on the *next* mutation to a shared chunk, but doesn't say what happens to reads of the shared chunk in the meantime — do both source and snapshot readers see identical bytes correctly before any copy-on-write has happened, or is there a subtler consistency wrinkle there?
- **source gap** ([[hdfs-block-placement-and-replication-management]]): the source describes the replication priority queue's two priority bands (single-replica chunks highest, >2/3-replicated lowest) but doesn't spell out how blocks are prioritized *within* a band, or how the background scanning thread paces itself against competing client traffic.
- Does S3's per-prefix throughput ceiling (3,500 PUT / 5,500 GET per second) still bind in practice once auto-splitting kicks in, or is the practical bottleneck now something else entirely? The source names the ceiling and the auto-split behavior but doesn't describe how the two interact under sustained hot-prefix load.
- **source gap** ([[s3-strong-consistency]]): the source names "a new component to check if the metadata cache is stale" but doesn't explain what that staleness check actually inspects (a version number, a timestamp, a direct compare against the persistence layer) — the same category of gap the kafka topic flags for KIP-405's analogous auxiliary state.
- With object storage's erasure coding and S3/GCS conditional writes now covering both durability and atomicity, what — if anything — would still justify choosing GFS- or HDFS-style whole-copy replication and coordinator-bound metadata for a *new* system built today? The posts describe the historical handoff (HDFS "handed over the crown") but don't argue the counterfactual.

## Synthesis
Three systems, one lineage, two very different eras of constraint. [[gfs]] and [[hdfs]] both start from the same 2000s-era bet: commodity hardware fails constantly ([[component-failure-as-normal]]), so build monitoring, replication, and recovery into the core rather than bolting them on, and centralize metadata in one RAM-resident coordinator because a single coordinator with global knowledge can make better placement decisions than a distributed one — as long as its *involvement* in the actual data path stays minimal ([[gfs-lease-and-write-pipeline]], [[hdfs-client-read-write-and-lease]]). Both durability ([[gfs-replica-management-and-fault-tolerance]], [[hdfs-block-placement-and-replication-management]]) and metadata persistence ([[gfs-metadata-and-operation-log]], [[hdfs-image-and-journal]]/[[hdfs-checkpointnode-and-backupnode]]) end up looking like close cousins between the two systems, down to nearly identical vocabulary. The RAM-bound coordinator that makes both systems tractable is also what eventually bounds them — [[hdfs-namenode-scaling-limit]] traces the concrete ceiling this hits in production, sourced from a different raw group's Uber case study rather than these papers themselves. [[amazon-s3]] is not a descendant of this design so much as a rebuttal to it: no coordinator at all, a flat prefix-partitioned keyspace distributed across 350+ microservices ([[prefix-as-folders]]), durability from erasure coding rather than whole-copy replication ([[object-storage-durability-erasure-coding]]), and a consistency story that evolved from eventual to strong by patching a metadata cache rather than by adopting lease-based mutation ordering ([[s3-strong-consistency]]). What S3 does *not* solve — multi-object atomicity — is handled the same way GFS handles concurrent appends: accept a weaker guarantee (per-object atomicity, [[s3-atomicity-and-conditional-writes]]) and let something built on top (Delta Lake, Hudi) construct the stronger guarantee it needs from that primitive. The historical throughline both posts converge on: HDFS was the data lake's default because it was the only credible option before cloud object storage matured; once S3/GCS existed with better economics and no operational burden, [[object-storage-as-backbone|object storage took over]], and everything from table formats to Kafka-compatible logs to Postgres itself has started rebuilding on top of that same disaggregated substrate rather than on a RAM-bound coordinator.
