## Comparisons
**The executor spectrum: Sequential → Local → Celery → Kubernetes.** These four aren't just increasingly powerful, they trade specific things away for specific things gained. [[sequential-executor]] buys pure simplicity (SQLite-compatible, no external deps) by giving up parallelism entirely, and by pausing the Scheduler while its one task runs — a flaw serious enough that Airflow 3 replaces it outright with [[local-executor]]. [[local-executor]] buys real concurrency (multiple processes, one machine) for the cost of a real database (MySQL/Postgres) and a hard ceiling: capacity is bounded by whatever the Scheduler machine itself can carry. [[celery-executor]] breaks that ceiling by decoupling workers from the Scheduler entirely, at the cost of a message broker, more components to run, and two very specific failure modes — the Noisy Neighbor problem (one heavy task starves its worker-mates) and underutilized workers sitting idle when traffic is low. [[kubernetes-executor]] goes furthest — a pod per task, its own image, its own resources — buying the best isolation of the four at the cost of per-task cold start and requiring real Kubernetes operational depth. Since Airflow 2.10, none of this is an all-or-nothing environment-wide choice: multiple executors can coexist, and Vu's own example puts a fast Celery queue on short numerous tasks and Kubernetes pods on long resource-intensive ones inside the very same DAG. See [[airflow-resource-isolation-strategies]] for how this executor-level choice composes with the finer, per-task escapes (`PythonVirtualenvOperator`, `DockerOperator`, `KubernetesPodOperator`, Celery queues) that don't require switching the whole environment's executor at all.

**Cron vs Sensors vs Assets vs message/API triggers.** All four answer "when should this run," but they sit on a spectrum from "we already know" to "we have to find out" to "we get told." Cron scheduling ([[airflow-scheduling-cron-vs-event-driven]]) is the default and is fine for most pipelines, but it carries no information about whether the data it's scheduled around is actually ready — a gap event-driven triggering exists specifically to close. [[sensors]] close it by polling (an S3KeySensor checking for a file, an ExternalTaskSensor checking another DAG's status) — an actively-waiting, poll-based strategy that is itself still just a task needing its own cron schedule. [[assets]] close it the opposite way: passively, by declaration — a task names the asset it updates via `outlets`, a downstream DAG names that asset in its own `schedule`, and Airflow does the triggering automatically the moment the producing task succeeds. The asset abstraction reaches further than files and tables: an `AssetWatcher` wrapping a `MessageQueueTrigger` lets a DAG react directly to an external stream like a Kafka topic, folding message-driven triggering into the same declarative shape rather than treating it as a fifth mechanism. The true fifth mechanism is Airflow's REST API, for cases needing more direct, imperative control than any of the above — an external system like an AWS Lambda function invoking a specific DAG run on demand.

**Dependency composition: bit-shift operators vs trigger rules vs TriggerDagRunOperator.** A pipeline's dependency graph is rarely a straight line — Vu names one-to-one, fan-out (one task feeding several in parallel), fan-in (several tasks converging on one), and cross-pipeline dependencies as the recurring shapes. Bit-shift operators (`>>`/`<<`) declare the graph itself, inside one DAG. [[trigger-rules]] decide *when* a task fires relative to its parents' outcomes — the default requires every parent to succeed, but `all_done` lets a task run regardless of whether its parents succeeded or failed, which matters because the alternative (exhausted retries triggering `upstream_failed` on everything downstream — see [[airflow-failure-handling-and-retries]]) is otherwise a hard skip, not something you can opt out of per-task without a trigger rule. [[trigger-dag-run-operator]] steps outside the single-DAG graph entirely, letting one DAG explicitly kick off a run of a completely different one — the imperative counterpart to declaring a dependency via [[assets|an Asset]]. [[xcom]] is the orthogonal piece that makes any of this useful in practice: a small mechanism for passing actual values (not just triggering signals) between tasks that are already connected by one of the above.

**Airflow's code-first flexibility vs Orchestra's UI-first abstraction.** [[orchestra|Orchestra]] targets the identical eight-category problem space Airflow does — scheduling, dependencies, triggers, environments, observability — but inverts the interface: instead of writing a Python DAG and custom Hooks, you configure managed "integrations" through a UI, and Orchestra's platform absorbs auth, error handling, triggering, polling, and metadata collection that an Airflow user would otherwise hand-roll. The gap Vu names concretely is dbt: Airflow ships a first-party operator only for dbt Cloud, forcing a custom operator for free/self-hosted dbt core, while Orchestra's dbt-core integration works immediately. Environments are the other clean contrast — Airflow, in Vu's account, needs separate infrastructure per environment (two full Spark clusters for dev and prod, say), while Orchestra treats an environment as pure configuration that the platform provisions and aligns behind the scenes. None of this is free: Vu's own verdict after building a real Snowflake+dbt pipeline on Orchestra is that it cannot match the flexibility of hand-written Airflow DAGs and custom operators — the trade is aimed squarely at teams who'd rather spend their limited engineering time on business logic than on maintaining an orchestrator.

**Backfilling and retries are both "run it again," but for opposite reasons.** [[backfilling]] re-runs history on purpose, because the *logic* changed (a bug fix that needs to be applied retroactively) — via the explicit `backfill` command or the `catchup` parameter at DAG-creation time. Retries ([[airflow-failure-handling-and-retries]]) re-run automatically because a *single execution* failed transiently — a down API, an overloaded source — with no change to the logic at all. Both depend on the same underlying safety net: a task scoped to its own `data_interval_start`/`data_interval_end` (or, for retries, simply re-attempted with the same inputs) is safe to run twice, which is exactly what task-level [[idempotency]] has to hold for either to be trustworthy. Both also stress the same concurrency knobs from the same direction — a 90-day backfill turning 10 daily tasks into 900 concurrent ones is exactly why `max_active_runs` needs a second look before backfilling (see [[airflow-concurrency-and-resource-control]]), just as a storm of simultaneous retries after a source recovers would.

## Open questions
- At what task volume does CeleryExecutor's Noisy Neighbor problem outweigh KubernetesExecutor's cold-start cost in practice? Vu names both trade-offs but gives no threshold or rule of thumb for choosing between them.
- With per-task executor assignment available since Airflow 2.10, what's the actual syntax or decision rule for assigning different executors to different tasks in one environment? The source states the capability and gives a queue-routing example for Celery, but doesn't show how a task is bound to a specific *executor* (as opposed to a specific Celery queue).
- **source gap** ([[trigger-rules]]): the source shows a table of "full options" for trigger rules via an image and details only the `all_done` example in prose — the other trigger rule options (e.g. one_failed, none_failed, none_skipped) are never spelled out in text.
- **source gap** ([[airflow-failure-handling-and-retries]]): the source names `DeadlineReference.DAGRUN_QUEUED_AT` as one deadline reference point but doesn't enumerate what other `DeadlineReference` options exist or when you'd pick them.
- **source gap** ([[assets]]): the Kafka-triggered DAG example elides the `MessageQueueTrigger`'s actual configuration (`queue=KAFKA_TOPIC, ...`) — consumer group, offset behavior, and delivery semantics for that trigger are never spelled out.
- **source gap** ([[airflow-concurrency-and-resource-control]]): the source names `weight_rule` values (downstream/upstream/absolute) and their intent but doesn't spell out how effective weight is computed when a task sits in more than one pool, or how ties are broken.
- How far does Orchestra's "metadata aggregation" go technically? Vu says native Snowflake tables are auto-collected while dbt models require explicitly opting each task in to collect metadata, but doesn't explain what differs mechanically between the two collection paths.
- Vu explicitly flags this himself as unresolved: how well does Orchestra support multiple domains and access governance in large enterprise orgs? He notes he didn't get to explore this fully, only that it's "very difficult using traditional workflow orchestration platforms" too — leaving the actual comparison open.

## Synthesis
Airflow's shape follows directly from its 2014 Airbnb origin ([[airflow-origin]]): workflows as versioned, programmable Python code rather than a rigid or non-scalable scheduler, built around Tasks and Dependencies as a DAG and run by five cooperating components — Scheduler, Web Server, Metadata Database, DAG folder, and Workers — cycling through define → parse → schedule → execute → monitor → retry on every run ([[airflow-core-components]]). Vu's own eight-category orchestration problem space ([[orchestration-problem-space]]) gives that architecture a job list, and each category resolves into concrete Airflow levers: Scheduling splits into cron's fixed intervals and event-driven triggering's polling ([[sensors]]) or declarative ([[assets]]) alternatives, extending as far as a Kafka topic or a direct API call ([[airflow-scheduling-cron-vs-event-driven]]); Dependency Management and Branching compose bit-shift operators, [[xcom]] for passing values, [[trigger-rules]] for status-conditional execution, [[trigger-dag-run-operator]] for crossing DAG boundaries, and [[conditional-operators]] for runtime routing; Failure handling layers retries, timeouts, and callbacks on top of a hard `upstream_failed` default, plus a separate SLA-miss alert for runs that are merely slow rather than broken ([[airflow-failure-handling-and-retries]]); Backfilling reprocesses history deliberately via `backfill`/`catchup`, leaning on the same interval-scoping (and task-level [[idempotency]]) that makes retries safe ([[backfilling]]); Concurrency & Resource Control nests global `parallelism` inside per-DAG `max_active_runs`/`max_active_tasks` inside per-resource [[pools]] inside priority-ordered queuing ([[airflow-concurrency-and-resource-control]]); and Resource isolation runs from the executor spectrum ([[sequential-executor]] through [[kubernetes-executor]]) down to per-task escapes that don't require an environment-wide executor change ([[airflow-resource-isolation-strategies]]). [[orchestra|Orchestra]] is the topic's one outside data point on all of this: it targets the identical problem space but answers it by trading Airflow's code-level flexibility for a managed, UI-configured platform — proof that the eight-category framing generalizes past Airflow itself, even as the concrete mechanics (DAGs-as-code vs. integrations-as-config) diverge completely.
