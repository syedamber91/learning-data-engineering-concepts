## Comparisons
**NSM vs DSM vs PAX — three answers to "where does a row's data live."** [[nsm]] keeps a row's columns contiguous inside a page, addressed by a slot array that grows toward the middle as row data grows from the other end — great for OLTP's fast single-row writes and reads, bad for compression since adjacent column values rarely share a pattern. [[dsm]] inverts this: each column gets its own pages, and a fixed-per-column value length turns "which row does this value belong to" into array arithmetic (`first_element_address + i * element_size`). That buys OLAP-friendly I/O (only the needed columns are touched) and much better compression, at the cost of expensive multi-column reads and writes — a 100-column insert means 100 separate page writes. [[pax-hybrid-layout]] splits the difference by partitioning horizontally into row groups first, then storing columns together *within* each group — Vu's central finding is that PAX, not true DSM, is what BigQuery, Snowflake, DuckDB, Parquet, and ORC actually run under their "columnar" branding; only [[clickhouse|ClickHouse]] and [[redshift|Redshift]] were found to implement genuine DSM.

**Access pattern follows storage layout, but isn't reducible to it.** [[oltp-vs-olap-access]] is the concept that ties storage back to the query shape each model was built to serve: OLTP is a map-like "find this one thing" problem, solved by the [[b-plus-tree-index|B+Tree index]] descending root-to-leaf in O(log n) disk reads; OLAP is a "summarize almost everything" problem, where even a perfect columnar layout still has to eliminate data at a finer grain than "which column" — which is exactly what Zone Map metadata, partitioning, and clustering exist to do. The B+Tree itself is a case study in why on-disk structures can't just be in-memory structures ported over: a balanced BST's pointer-chasing costs up to 20 non-contiguous disk seeks for one lookup, while a B+Tree's wide, page-aligned nodes turn the same lookup into a handful of block reads — the entire reason B+Trees dominate OLTP indexing, and the entire reason they're *not* the OLAP answer (a lookup index doesn't help when you're scanning billions of rows for a handful of columns).

**Partitioning, clustering, and Z-ordering as increasingly fine-grained elimination.** [[partitioning-schemes-across-systems]] shows partitioning is implemented differently almost everywhere: BigQuery treats a partition as its own virtual table, ClickHouse manages each partition as an independent unit, Snowflake refuses to expose a partition key at all in favor of automatic 50–500MB [[snowflake-micro-partitions]], Redshift supports no user partitioning whatsoever, and Hive-derived systems (plus Delta/Iceberg/Hudi on top) fall back to directory-per-partition with progressively richer metadata layered on. [[clustering-sort-order-and-z-ordering]] operates one level finer, inside whatever partitions exist: plain sort-clustering is cheap but column-order-dependent (a filter on a trailing sort column gets no pruning benefit), while Z-ordering's bit-interleaved Morton codes trade a little of sort-clustering's best case for a lot less of its worst case, treating all clustered dimensions more evenly. Both partitioning and clustering are really in service of the same lower-level mechanism [[oltp-vs-olap-access]] names — Zone Map min/max statistics — since an unclustered, unpartitioned block's min/max range covers almost everything and prunes nothing.

**The PAX-vs-real-time-write tension.** Column-major layouts like [[pax-hybrid-layout|PAX]] are expensive to write into directly at high frequency, which is why BigQuery and Apache Hudi both land on the same fix Vu found independently in each: ingest into row-store format first, then convert to columnar/hybrid format in the background for read-side consumption. That's a genuinely different problem from clustering's write-vs-read trade-off — it's about *format* conversion timing rather than physical ordering — but it stems from the same underlying fact that column-oriented physical layouts are optimized for scan-heavy reads at the expense of point-by-point writes.

## Open questions
- Vu found only BigQuery and Apache Hudi using the row-store-then-background-convert-to-columnar pattern for real-time OLAP ingestion, and says so is due to "information limitations" — the posts don't say whether other OLAP systems (Snowflake, ClickHouse, Redshift) use an equivalent technique for their own real-time ingestion paths, or handle high-frequency writes into PAX/DSM storage some other way entirely.
- **source gap**: the posts describe DSM's offset arithmetic (`first_element_address + i * element_size`) for locating a value within one column's pages, but don't spell out how the DBMS locates the *page itself* on disk for a given column, or how a query recombines offsets across many columns once each column may be split across multiple pages.
- **source gap**: Redshift is named as not supporting user-defined partitioning, but the posts don't explain what mechanism (if any) Redshift uses in its place to prune data at partition-like granularity, given it is also one of only two true-DSM systems the posts identify.
- **source gap**: the posts name Iceberg, Delta Lake, and Hudi as all supporting Z-ordering, but don't spell out whether their Z-ordering implementations differ from each other, or how Z-ordering interacts with Iceberg's own hidden-partitioning/partition-evolution metadata (see [[iceberg-hidden-partitioning-and-sort-order]]) when both are applied to the same table.
- **source gap**: Vu explicitly skips the Hilbert curve as an alternative space-filling curve for multi-dimensional clustering, admitting he didn't want to work through the math — so this pack has no grounded account of how Hilbert-curve clustering compares to Z-ordering in practice, only that it exists and "tries to do the same things."
- The posts describe B+Tree overflow/underflow and WAL-based durability in detail for OLTP, but don't connect this back to how an OLAP system handles mutation of already-columnarized PAX/DSM data (e.g. deletes or updates against MergeTree in ClickHouse, or copy-on-write vs merge-on-read in table formats) — that mutation story is left entirely to other topics ([[clickhouse-internals]], [[iceberg]]) rather than resolved here.

## Synthesis
Take a table and ask where a given row's bytes actually live, and the honest answer is a spectrum, not a binary choice. [[nsm]] keeps a row whole inside one page, addressed by a slot array — ideal when the workload is "insert/update/read one record fast," which is the OLTP norm. [[dsm]] splits every column into its own pages and reconstructs rows via fixed-length offset arithmetic — ideal when the workload is "scan a few columns across everything," which is the OLAP norm, but genuinely rare in its pure form: only ClickHouse and Redshift were found to implement it. Nearly everything else marketed as "columnar" — BigQuery, Snowflake, DuckDB, Parquet, ORC — is [[pax-hybrid-layout|PAX]], a compromise that partitions horizontally into row groups first and stores columns together only within each group, trading a little of DSM's compression and I/O advantage for much cheaper row reconstruction. That storage choice in turn dictates how each side finds data: OLTP's [[b-plus-tree-index|B+Tree]] descends a shallow, disk-block-aligned tree to answer "where is this one row," while OLAP layers three coarse-to-fine elimination techniques on top of columnar storage — Zone Map metadata, [[partitioning-schemes-across-systems|partitioning]] (implemented differently in nearly every system, from BigQuery's virtual-table partitions to Snowflake's automatic micro-partitions to Redshift's outright lack of support), and [[clustering-sort-order-and-z-ordering|clustering]] (plain sort, or Z-ordering's fairer multi-dimensional variant) — all in service of the same goal [[oltp-vs-olap-access]] names as OLAP's defining problem: not finding a row, but avoiding reading the ones you don't need. The throughline across every post: the label a vendor puts on its storage engine ("columnar," "partitioned," "clustered") is worth less than knowing which of these concrete mechanisms it actually implements.
