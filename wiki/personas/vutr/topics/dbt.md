## Comparisons
**SCD Type 1 vs Type 2 vs Type 3 — a real preference ranking, not a neutral menu.** Vu presents all three as live options but doesn't treat them as equally good defaults. [[scd-type-1-and-3|Type 1]] (overwrite) is the cheap, history-destroying baseline — fine only when the past genuinely doesn't matter, and his own worked example (a user moving Vietnam→Singapore) shows exactly what breaks: the earlier country's sales retroactively vanish from a country breakdown. [[scd-type-2]] is what he calls the most-used technique and treats as his own default, because inserting a new row with `effective_date`/`expired_date` (and an `is_current` flag for cheap lookups) preserves history without ever touching the schema. [[scd-type-1-and-3|Type 3]] (extra columns for a prior value) is the one he's openly skeptical of — he can't think of a real use case beyond its narrow "observe two dimension versions at once" theoretical justification, and he passes along the claim that a `LAG` window function over an ordinary Type 2 table reproduces its effect on modern SQL engines, making Type 3 largely redundant now that window functions are standard and disk is cheap.

**The hybrid types (4 through 7) are documented for completeness, not recommended.** [[scd-type-4-mini-dimension]] is the one entry in this whole SCD family where Vu found the industry genuinely divided: the book's own mini-dimension approach (a separate table storing only *unique combinations* of fast-changing attributes, banded to keep row counts small) versus a second "approach B" most online sources describe instead (current-state table plus a separate history table). He couldn't find an official reason the two diverged and chose to present both rather than force a single canonical definition. [[scd-hybrid-types-5-6-7]] builds upward from there — Type 5 bolts a Type 1 "outrigger" column onto the base dimension so current mini-dimension values are readable without a second join; Type 6 (whose name is literally a joke about 1+2+3=6=1×2×3) stores full Type 2 history while also carrying Type 1's overwritten current value and Type 3's as-was value in every row; Type 7 splits Type 6's single table into separate current/history tables keyed by a durable entity identifier. Vu is explicit that he doesn't believe these are widely adopted in practice — they're in the wiki because he researched them, not because he'd reach for them.

**ETL vs ELT is an economics story, not a syntax swap.** [[elt-vs-etl]] is Vu's flat rejection of the idea that ELT is just ETL with the last two letters reordered. ETL made sense specifically because 1970s-through-2000s warehouses were storage- and compute-constrained, so you had to curate a small subset before loading it — and that constraint is also why ETL pipelines were slow, DE-gatekept, and brittle to changing requirements (no raw source data survived downstream to re-derive from). ELT became viable only once cloud warehouses made storage and elastic SQL compute cheap enough that keeping *all* the raw data and transforming later, in SQL, stopped being wasteful. Vu is careful not to overclaim the reversal: he states plainly that ELT will not fully replace ETL and that some cases still require it, without naming which ones.

**dbt is a transformation tool; a data model is a design decision — and conflating them is the single claim Vu repeats most insistently across this whole group of posts.** [[data-modeling-is-not-dbt-modeling]] is the explicit statement: a data model is tool-agnostic and defines how data is structured and related, while a dbt model is a SQL script that shapes data inside the warehouse. He makes the same point three separate ways: writing dbt models is not modeling (his `why-is-dbt-so-popular` research); staging his own project through medallion-style landing/staging/curated layers is not modeling either, it's just a way to facilitate cleaning (his hands-on Kimball project); and dumping data into [[one-big-table]] only pays off with a real modeling layer already underneath it, otherwise you've traded understandability for query speed and called it a win. [[dimensional-modeling]] is what he holds up as the actual modeling discipline dbt gets used to implement — Kimball's four-step process (business process → grain → dimensions → facts) done correctly is orthogonal to, and prior to, any tool that executes the resulting SQL.

**dbt's rise and [[democratization-of-transformation]] are the same causal chain told from two angles.** [[dbt-origin-and-adoption]] is the outside view: Tristan Handy built dbt in 2016 to bring software-engineering discipline (version control, testing, modularity) to SQL transformation, adoption went from 3 companies to 9,000+ by 2022, and Vu explains *why* it won using the UTAUT framework — low effort to learn, strong social proof, minimal infrastructure prerequisites. [[democratization-of-transformation]] is the inside view of the same shift: a data engineer can manage two or three pipelines well but becomes the bottleneck past fifty, and letting SQL-literate data analysts write transformation logic directly — rather than queueing behind a DE for every request — lets datasets reflect business needs from the start. Vu is explicit that this collaboration model doesn't eliminate DEs; it reassigns them to the harder engineering judgment (query optimization, macro standards, infrastructure) underneath the SQL analysts now write themselves. Both origin stories converge on the same precondition named in [[elt-vs-etl]]: none of this works without the cloud warehouse making in-warehouse SQL transformation viable and affordable in the first place.

**Two ways to actually build SCD Type 2 inside dbt — and Vu changed his mind about which to recommend.** [[dbt-incremental-strategies-for-scd]] is his own head-to-head: hand-writing the Type 2 logic through dbt's `merge` incremental strategy (comparing incoming rows against only the `is_current = true` slice of the target table, detecting new-vs-changed rows by [[surrogate-keys|surrogate key]] mismatch, explicitly expiring superseded rows) versus dbt's built-in `snapshot` feature, which most external resources recommend and which he initially skipped specifically to force himself to learn the mechanics the hard way. Having built both, his production recommendation flips to snapshot — it's the default in the sense of "let dbt handle the bookkeeping" — even though snapshot's requirement that the tracked table be a registered dbt `source` forced an awkward workaround in his own project (materializing a temp model, then registering *that* as a source, purely to make it snapshot-eligible). The fact table itself needed a third, different strategy again — `insert_overwrite`, chosen specifically for idempotency, since `append` would silently duplicate rows on a rerun.

## Open questions
- Which specific cases still make ETL necessary rather than ELT, now that transformation has largely moved inside the warehouse? Vu says such cases exist but never names them ([[elt-vs-etl]]).
- How far can [[democratization-of-transformation]] via dbt go before the absence of a real data model starts to hurt — where is the line between enabling analysts and quietly reintroducing the mess a data model is supposed to prevent? Vu names the tension in two separate posts ([[democratization-of-transformation]], [[data-modeling-is-not-dbt-modeling]]) but never draws the line.
- **source gap** ([[scd-type-4-mini-dimension]]): Vu documents that SCD Type 4 has two genuinely competing definitions in circulation (mini-dimension vs. current/history split) and says there's "no official reason" for the divergence — which one, if either, traces back to Kimball's original intent is left unresolved.
- Vu could not find, and could not himself construct, a convincing real-world case for SCD Type 3 beyond the theoretical "observe two dimension versions simultaneously" scenario ([[scd-type-1-and-3]]) — whether one genuinely exists in production is left open.
- **source gap** ([[dbt-incremental-strategies-for-scd]]): Vu names `insert_overwrite` as idempotent for the fact table (in contrast to `append`, which duplicates) but doesn't spell out the underlying mechanism — e.g. whether it's a partition-replace operation and how the affected partition boundary is actually determined.
- Vu explicitly invites correction on his own dbt snapshot implementation, calling the "register a materialized dbt model as a dbt source just to make it snapshot-eligible" pattern a possible workaround rather than a clean use of the feature ([[dbt-incremental-strategies-for-scd]]) — whether there's a cleaner idiom for multi-table SCD 2 sources is left to the reader.
- Vu titles his own dbt-popularity research with a question mark — "becoming a transformation standard(?)" — and never states outright whether he considers that standardization complete or still underway ([[dbt-origin-and-adoption]]).

## Synthesis
Read together, these seven posts are Vu working through Kimball dimensional modeling and dbt as two halves of one discipline that keep getting confused for each other. [[dimensional-modeling]] supplies the design method — business process, then grain ([[grain-declaration]]), then dimensions, then facts, assembled into a [[star-schema]] deliberately traded against 3NF for business-user legibility — and [[surrogate-keys]] plus [[scd-type-2]] (with [[scd-type-1-and-3|Type 1, Type 3]], and the [[scd-type-4-mini-dimension|Type 4]] / [[scd-hybrid-types-5-6-7|Type 5-7]] hybrids as the fuller catalogue of ways to track change) are the mechanics that make that model track a business that never stops changing underneath it. [[dbt]] and its [[dbt-origin-and-adoption|origin and adoption]] story are the tool, not the method: it exists because [[elt-vs-etl|ELT's]] cloud-warehouse economics made in-warehouse SQL transformation cheap enough to need version control, testing, and modularity, and it succeeded because it gave that discipline to SQL-literate analysts without requiring a data engineer to write every line — the [[democratization-of-transformation]] Vu experienced firsthand starting in 2021. But Vu's own hands-on project — building `dim_product` and `dim_territories` through both a hand-rolled `merge` and dbt's `snapshot` feature ([[dbt-incremental-strategies-for-scd]]), then a fact table via `insert_overwrite` — is also where he draws the line he keeps insisting on: dbt shapes data, it does not decide how that data should be structured and related. A medallion-style landing/staging/curated layering, a [[one-big-table|One Big Table]], or a tool as capable as dbt can only ever sit on top of the modeling decisions in [[dimensional-modeling]] — and per [[data-modeling-is-not-dbt-modeling]], a layer thrown over an absent model just yields another mess, no matter how well the SQL managing it is engineered.
