## Comparisons

**LSM-tree vs B-Tree: two answers to the same in-place-update problem.** A B+Tree-based engine writes a 30-byte key-value pair by reading the full 4KB/8KB page containing it, modifying those bytes in memory, and writing the *entire* page back to disk — high [[write-amplification-tradeoff|write amplification]] regardless of how small the actual change was. The [[b-tree|B-Tree]] pays that cost on every random write in exchange for a genuine advantage: reads follow a single, predictable `O(log n)` path from root to leaf, which is why Vu calls it ideal for OLTP workloads needing low-latency random access. The LSM-tree refuses that trade in the other direction — both engines pay the same [[write-ahead-log]] cost for durability, but the LSM-tree's heavy I/O is a *sequential* flush of an already-sorted [[memtable]] rather than an in-place page overwrite (see [[sequential-vs-random-io]]), which is what lets it sustain higher write throughput than a B-Tree in most cases. The cost shows up on the read side instead: an LSM-tree read may have to check the Memtable and then several on-disk [[sstable]] files across levels before it finds — or fails to find — a key.

**Size-Tiered vs Leveled compaction: two ways to spend the same rewrite budget.** Within [[compaction]], the two strategies sit at opposite ends of the read/write amplification spectrum. Size-Tiered (Apache Cassandra) groups SSTables by similar size and only merges — and rewrites — when a tier accumulates enough files to be promoted; it's write-optimized, but because its trigger is file count rather than key range, SSTables within a tier can hold overlapping keys, forcing reads to check multiple overlapping files to find a key's latest version (high read amplification). Leveled compaction keeps all levels but L0 non-overlapping, so reads need fewer checks — but pulling even a small amount of new data into a level forces a much larger volume of unrelated, overlapping data already in that level to be rewritten, making it the more write-amplifying of the two. Neither strategy is free; they just decide which side of the read/write tax the engine pays.

**The read path's two shortcuts stack, they don't compete.** A [[sstable]]'s sparse index (one entry per block, e.g. every 8,000 rows) narrows a lookup to a range and still requires a binary search within it — it can tell you *where to look*, never *whether to bother looking*. The [[bloom-filter]] answers that second question instead: per-SSTable, it says "definitely not here" with zero false negatives, letting the system skip opening the file entirely, or "maybe here," in which case the sparse index takes over. The two mechanisms are complementary layers of the same read path, not alternatives to each other.

**LSM reappearing in OLAP, at three different depths of commitment.** [[bigquery-vortex|BigQuery Vortex]] applies the LSM idea to the *main data path*: every Fragment ages from a Write-Optimized-Storage hot format into a Read-Optimized-Storage cold columnar format via a background Storage Optimization Service — Memtable-flush-then-compact, just relabeled at OLAP scale. Apache Hudi applies the same layered-tree idea only to its *metadata*: its original one-small-file-per-table-action Timeline design hit a small-file problem on cloud storage as tables aged, and Hudi 1.0's LSM Timeline fixed it by batching timeline metadata into a layered L0/L1/L2 tree of larger Parquet files — the LSM pattern solving a bookkeeping problem, not a data-ingestion one (see [[hudi-timeline]] for the Timeline's own action/state mechanics, which predate this LSM batching fix). ClickHouse's MergeTree family sits a level shallower still: Vu calls it merely LSM-*inspired* — data lands in "parts" stored in primary-key sort order, each part storing its columns independently, later merged in the background into larger parts — the LSM shape (write buffer → immutable segments → background merge) without necessarily being a literal Memtable/SSTable/WAL implementation (see [[clickhouse-internals]] for the actual mechanics: MergeTree's parts, granule-addressed marks, and merge heuristics).

## Open questions
- What concrete threshold triggers a Memtable flush? Vu is explicit that the definition of "full" is system-dependent and configurable, but the post never names a rule of thumb (size, row count, or time-based) for any specific engine.
- Beyond "write-optimized" vs "read-optimized," what actually determines a real system's choice between Size-Tiered and Leveled [[compaction]] — workload shape, hardware, something else? The post states the trade-off but not the decision procedure.
- How does compaction resolve [[tombstone]]-marked deletes specifically under Size-Tiered's overlapping-tier structure, where the same key can legitimately exist in several tiers at once?
- The post flags its own uncertainty on generality: "not all implementations of LSM in OLAP systems utilize different file formats for other purposes... this is based solely on my research" — so how common is the row-in-memory-then-columnar-on-compaction pattern actually is across LSM-based OLAP engines remains unresolved.
- Which real-time OLAP systems specifically *don't* implement the LSM-tree, and why not? Vu's disclaimer says not all of them do, but the post never names one or explains the alternative.
- How does [[bigquery-vortex]]'s Storage Optimization Service decide a Fragment is "cold" enough to convert from WOS to ROS — by size, age, access pattern? The post describes the two-format pipeline but not the trigger.

## Synthesis
Vu's account of the LSM-tree centers on one inversion: instead of updating data in place like a [[b-tree]], keep an in-memory, *sorted* — not append-only — [[memtable]] (Skip List, Red-Black Tree, or AVL Tree), guard it against crashes with a [[write-ahead-log]], and when it fills, flush it in one sequential pass to an immutable, sorted [[sstable]] on disk. That single design choice — sequential flush over in-place page rewrite (see [[sequential-vs-random-io]]) — is the entire source of the LSM-tree's write-throughput advantage, and the entire source of its cost on reads, which must check the Memtable and then potentially many SSTables across levels. Two shortcuts keep that read cost tolerable: a sparse index narrows the search within any given SSTable, and a [[bloom-filter]] skips whole SSTables outright when a key is provably absent. Updates and deletes never touch old data directly — updates write a new versioned key, deletes write a [[tombstone]] — and the actual cleanup, plus the merging of small flushed files into fewer larger ones, happens in the background via [[compaction]], whose Size-Tiered and Leveled strategies trade write amplification against read amplification in opposite directions. The reason this matters beyond OLTP, per Vu, is that OLAP systems increasingly need the LSM-tree's write-throughput strength to absorb continuous streams (Kafka, CDC, IoT logs) while still answering queries in near-real-time — and they adopt the pattern at different depths, from [[bigquery-vortex|BigQuery Vortex]]'s full WOS→ROS data path, to Hudi's metadata-only LSM Timeline, to ClickHouse MergeTree's looser, "inspired by" borrowing of the same shape.
