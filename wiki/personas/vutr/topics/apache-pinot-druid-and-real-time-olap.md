## Comparisons
**Two lambda architectures, drawn at different layers.** Both [[apache-pinot]] and [[apache-druid]] answer the OLTP-can't-do-this / classic-OLAP-can't-do-this gap the same way in spirit — split fresh data from durable data — but draw the split in different places. Pinot draws it as two entire systems: Kafka feeds the online/real-time half directly, Hadoop feeds the offline/global half, and a query spanning both gets rewritten into two sub-queries at a time boundary and merged ([[offline-realtime-query-merge]]). Druid draws the same split *inside* its own node types instead of standing up a separate batch system: real-time nodes serve just-ingested data from an in-memory buffer ([[druid-realtime-node]]), while historical nodes serve the durable, deep-storage-backed segments once that data has been finalized ([[druid-historical-node]]). Neither system treats "real-time OLAP" (see [[real-time-olap]]) as one code path — both bake the fresh/durable distinction into their architecture, just at different granularities.

**A shared bet on immutability, two different roads to it.** Both engines store data as immutable, columnar segments (see [[immutable-segment]]), and both cash in the same payoff: consistency and parallelism without read-time coordination. But they get there differently. Pinot's segments are fixed, replicated, read-optimized units from the moment they're written — any update produces a *new* segment. Druid's segments are the end product of a conversion: an in-memory row-store buffer on the real-time node gets periodically flushed to columnar form on disk, merged into a time-bounded block, and only then shipped to deep storage as an immutable segment. Getting there from the "fresh, mutable" side (Druid) versus starting immutable and replicated from the start (Pinot) is a genuinely different engineering path to the same read guarantee.

**Consensus for correctness shows up in only one of the two sources — and it's a hard problem.** Pinot's segment-completion protocol ([[segment-completion-protocol]]) exists because independent segment replicas consuming the same Kafka offsets can still diverge (clock drift), so the controller arbitrates via a six-instruction state machine (HOLD/DISCARD/CATCHUP/KEEP/COMMIT/NOTLEADER) to guarantee every replica ends up bit-identical once flushed. The Druid source describes an analogous problem space — real-time nodes persisting in-memory buffers to disk, tracking Kafka offsets for recovery — but never describes an equivalent multi-replica reconciliation mechanism; it only distinguishes redundant consumers (same events, for fault tolerance) from load-balanced consumers (different partitions, for throughput). Whether Druid needs an equivalent protocol for its own redundant-consumer case, or avoids the problem some other way, isn't something either source resolves (see Open questions).

**Query routing: scatter-gather-merge either way, but different fault and tenancy postures.** Pinot's broker ([[pinot-broker]]) pre-generates routing tables and offers two routing strategies — a default balanced strategy that spreads segments evenly (fine until a cluster gets large enough that any query has good odds of hitting a struggling server) and a dedicated large-cluster strategy that deliberately limits how many hosts a single query touches, to cap the blast radius of a bad host. It also runs token-bucket multi-tenancy so no tenant can starve others. Druid's broker ([[druid-broker]]) instead centers its design on a caching asymmetry: it caches historical-node results per segment but *never* caches real-time-node results, trading away that caching opportunity specifically to guarantee every real-time-touching query is always answered fresh. Neither broker eliminates dependence on its cluster metadata store (Zookeeper for both) — Druid's broker explicitly falls back to its last-known cluster state on a Zookeeper failure and assumes nothing has changed since.

**Query-surface trade-off vs. structural indexing.** Pinot narrows its query surface deliberately — [[pinot-pql]] drops joins, nested queries, DDL, and record-level operations — and recoups performance by layering structural indexes on top of immutable segments instead: the [[star-tree-index]] answers iceberg-style aggregate-and-threshold queries from pre-aggregated tree nodes without touching raw records, alongside range and bitmap inverted indexes. Neither post directly benchmarks Pinot against Druid on latency or resource cost — that comparison isn't something the sources make, and isn't claimed here.

## Open questions
- Does Druid have anything analogous to Pinot's segment-completion protocol for reconciling redundant real-time consumers reading the same Kafka events, or does it resolve replica divergence some other way? The Druid source never says.
- Pinot's PQL forgoes joins and nested queries entirely — what do teams do instead when a use case would otherwise need one? The source states the limitation but not the workaround.
- The Druid source is explicitly based on a 2014 paper and flags that later components (like Overlord nodes) aren't covered — how much of the real-time-node/historical-node/broker/coordinator picture described here still matches Druid's current architecture?
- Pinot's routing-table selection picks a routing table "based on a specific metric" among several generated candidates — the source names that this scoring happens but not what the metric actually is.
- Both papers describe historical/servers reading from deep storage/object storage as a cache-miss path, but neither spells out what happens to query latency or correctness during a cold start when a node's cache is empty across many concurrent queries.

## Synthesis
[[apache-pinot]] and [[apache-druid]] are two production answers to the same requirement — analytical queries at low latency and high QPS, which neither classic OLTP nor classic batch-oriented OLAP can deliver (see [[real-time-olap]]) — arrived at independently by LinkedIn and Metamarkets. Both bet everything on immutable, columnar segments as the storage unit ([[immutable-segment]]), and both split fresh data from durable data rather than serving everything from one path, just at different granularities: Pinot as two whole systems reconciled at query time ([[offline-realtime-query-merge]]), Druid as node-type specialization within one cluster ([[druid-realtime-node]], [[druid-historical-node]]). Where they diverge is in how each protects correctness and speed once that bet is made: Pinot builds an explicit controller-arbitrated consensus protocol to keep Kafka-consuming replicas identical ([[segment-completion-protocol]]), narrows its query language to stay fast, and recovers expressiveness through structural indexes like the star-tree ([[star-tree-index]]); Druid protects freshness through a caching policy at the broker rather than a replica-reconciliation protocol, and tiers its historical nodes by access frequency rather than layering additional indexes. Neither source ever states which engine is faster or cheaper in general — that comparison, tempting as it is, isn't one the two posts actually make.
