## Comparisons
**Three systems, one substrate, three different problems.** [[dremel-query-engine]], [[vortex-storage-engine]], and [[napa-serving-system]] all sit on Colossus (storage) and Borg (compute orchestration), with Spanner backing whichever one needs strict transactional metadata — yet each exists because the other two don't fit its workload. Dremel is built for ad hoc, minutes-to-hours analytical SQL over petabytes it may never have seen before. Vortex is built to let that same warehouse ingest streaming and batch data with the same abstraction (a Stream, in three flavors — UNBUFFERED/BUFFERED/PENDING — mapping directly onto how strict the caller needs durability to be). Napa exists because neither of the other two can hit sub-millisecond, high-freshness, high-throughput serving for Google's own billion-user products — it reuses F1 Query and materialized views instead of Dremel's dynamic query plan, precisely because its queries are simpler and its latency budget is far tighter. Vu's own read is that Vortex is the unusual member of this trio: Google built it specifically because BigQuery's legacy batch storage engine could not be stretched into a real-time system, whereas Napa was apparently designed for real-time from the outset.

**Two-format storage shows up twice, tuned for different trade-offs.** Vortex's WOS→ROS split and Napa's WO/RO split are the same underlying idea — separate a write-optimized format from a read-optimized one and let a background process convert between them — but the systems dial the trade differently. Vortex's conversion unit is the Fragment, chosen small enough that WOS→ROS conversion happens often (keeping data close to read-ready) but not so small that a Stream Server drowns in Fragment metadata; Napa instead exposes the WO/RO conversion cadence as an explicit lever clients can tune along freshness/performance/cost, because Napa's clients (internal Google services) have wildly different tolerances for staleness than a BigQuery analyst running an ad hoc query does. Both ultimately compact into a Capacitor-like PAX/columnar read format — Vu explicitly calls out Napa's RO format as sharing Parquet/BigQuery/Snowflake's PAX-like character (see [[capacitor-file-format]]).

**Disaggregation is a repeated move, not a one-time architectural choice.** Dremel disaggregated twice in its own history, for different reasons and years apart: storage moved off local disks onto GFS around 2009–2012 to escape the load-time and independent-scaling penalties of shared-nothing servers (see [[storage-compute-disaggregation-bigquery]]), and the shuffle layer was separated from compute afterward to escape the quadratic-scaling and resource-fragmentation costs of coupling shuffle's temporary storage to the worker (see [[disaggregated-shuffle-service]]). Both moves accept an up-front cost (GFS's initial "order-of-magnitude" latency regression; the extra hop of writing shuffle data over the network instead of to local disk) in exchange for the same underlying prize: storage/memory that scales independently of compute, and — for shuffle specifically — a checkpoint that lets the scheduler treat workers as stateless and resize stages mid-query.

**Two unrelated mutation stories, from two different eras of source material.** [[immutable-storage-sets-and-dml]] (drawn from the 2019 BigQuery book) describes INSERT/UPDATE/DELETE as whole-new-immutable-files wrapped in ACID storage sets with a PENDING→COMMITTED→GARBAGE lifecycle. [[vortex-storage-engine]] (drawn from Google's 2024 Vortex paper) describes mutation differently: a virtual `_CHANGE_TYPE` column (INSERT/UPSERT/DELETE) and deletion masks applied to Fragments, with a special "tail" case for rows written since the last heartbeat. Neither source explains how — or whether — these two mechanisms are actually the same system described at different levels of abstraction, or two genuinely different code paths (e.g., batch-loaded/DML-query tables vs. Vortex-streamed tables) that BigQuery runs side by side; see Open questions.

**Two different "make the metadata columnar" moves, one on data structure and one on the catalog.** [[definition-and-repetition-levels-dremel]] and [[big-metadata-cmeta]] both trade extra bytes for I/O savings by pushing a columnar mindset one level down from where it normally applies. Definition/repetition levels embed a record's ancestor-presence structure directly into each column value so a nested field can be read without ever opening its ancestors — at the cost of redundant structural bytes on every child field. CMETA instead stores a table's own *metadata* (block locations, statistics) as a columnar table in its own right, so a query only pays for the metadata columns (e.g. `_block_locator`, a timestamp's min/max) it actually references, rather than reading a monolithic block header per file. Both bets pay off because the underlying engine (Dremel) can push a real distributed query — not a linear scan — through either kind of columnar structure.

## Open questions
- **source gap** ([[disaggregated-shuffle-service]]): the Dremel paper doesn't detail how the separated "distributed transient storage system" is actually implemented; Vu had to supplement it from an external talk, and even that supplement stops at high-level benefits (independent scaling, fault tolerance, flexible worker counts) rather than mechanism.
- **source gap** ([[disaggregated-shuffle-service]]): Vu names Uber's and Facebook/Meta's Spark shuffle-separation work as analogous but only links to their own external engineering posts — no first-hand detail on those systems' mechanisms is captured in vutr's own writing.
- **source gap** ([[big-metadata-cmeta]]): the source describes converting a query's `WHERE` filters into "derivative expressions on column properties" for CMETA pruning, but explicitly defers the actual framework to the paper rather than explaining it.
- **source gap** ([[immutable-storage-sets-and-dml]]): the storage optimizer's compaction is described only as running "automatically...behind the scenes" — no detail on what triggers a compaction pass or how candidate files are chosen.
- **source gap** ([[vortex-storage-engine]]): the source describes the SMS mapping a provisionally-deleted Streamlet "tail" down to real deletion masks once the next heartbeat reports the tail's actual Fragments, but doesn't spell out that mapping algorithm.
- Do the storage-set/immutable-file DML model ([[immutable-storage-sets-and-dml]], sourced from the 2019 book) and Vortex's `_CHANGE_TYPE`/deletion-mask model ([[vortex-storage-engine]], sourced from the 2024 paper) describe the same underlying mutation path at different points in BigQuery's history, or two mechanisms that coexist for different ingestion routes (batch/DML query vs. streamed Vortex writes)? Neither source vutr captured reconciles the two directly.
- Vu flags Napa's own read as still incomplete: how the on-the-fly aggregation at serving time (when LSM compaction hasn't caught up) actually merges scattered updates for a single record is asserted but not walked through mechanically.

## Synthesis
Across vutr's nine BigQuery posts, the same story repeats at every layer he examined: Google keeps finding a place where one resource (shuffle's temporary storage, table storage, metadata, a write path) is tightly coupled to compute, and keeps pulling it apart into its own independently-scaled service — Colossus/Borg underneath, a disaggregated [[disaggregated-shuffle-service|shuffle layer]] and [[storage-compute-disaggregation-bigquery|GFS-backed storage]] inside [[dremel-query-engine|Dremel]] itself, a purpose-built streaming-first storage engine ([[vortex-storage-engine|Vortex]]) sitting beside the older [[immutable-storage-sets-and-dml|immutable-storage-set DML model]], a metadata system ([[big-metadata-cmeta|Big Metadata/CMETA]]) that is itself queried like data rather than loaded like an index, and a whole separate low-latency serving system ([[napa-serving-system|Napa]]) built because neither Dremel's dynamic-but-heavyweight plans nor a traditional warehouse's freshness could serve Google's own billion-user products. The physical encoding underneath all of it — [[capacitor-file-format|Capacitor]]'s columnar layout, [[definition-and-repetition-levels-dremel|definition and repetition levels]] for nested data, [[partitioning-and-clustering-bigquery|partitioning and clustering]] for pruning — is the constant that every one of these disaggregated services is ultimately built to serve efficiently. The throughline Vu keeps landing on: BigQuery's "just type SQL" simplicity is a facade over a system that never stopped re-litigating which resource should be separated from which, trading engineering complexity (heartbeats, tails, compaction, checkpoints) for the ability to scale each piece on its own terms.
