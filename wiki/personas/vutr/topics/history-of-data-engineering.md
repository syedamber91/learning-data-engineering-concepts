## Comparisons
**Two forks off the same relational trunk.** Codd's 1970 relational model gave the field SQL and the table as its basic unit; everything downstream of it in this history is a variation on how to organize and move data through tables at scale. The first fork is [[kimball-vs-inmon]]: Inmon's late-1980s "subject-oriented, integrated, nonvolatile, time-variant" definition of the warehouse and his top-down build-the-EDW-first design, against Kimball's 1996 bottom-up, marts-first dimensional modeling. The two never resolved into one winner — they're presented as two named, competing philosophies, not a solved debate. Inmon's own ETL origin story is the quieter half of this fork: he found hand-written extract/transform/load scripts repetitive enough that automated tooling (Informatica, IBM DataStage, later Cognos and SSIS) became necessary, which is the same underlying problem — moving data into a place built for questions rather than transactions — that both Kimball and Inmon were solving from opposite ends.

**MapReduce's low-level cost vs Hive's SQL patch.** [[hadoop-mapreduce]] became the big-data workhorse off the back of Google's GFS (2003) and MapReduce (2004) papers, but it is a genuinely low-level model — users must explicitly write Map and Reduce tasks. [[apache-hive]] existed specifically to patch that: a SQL-like language (HiveQL) that Hive's Compiler turns into a DAG of MapReduce jobs, plus a table abstraction (databases, tables, partitions, buckets mapped onto HDFS folders) and a Metastore catalog so a SQL user has a `FROM` clause to reach for at all. The patch worked well enough that Hive became the widely-adopted default — until the same low-level cost that justified Hive's existence became irrelevant, because MapReduce itself stopped being the execution engine anyone wanted.

**Hive's two halves aged differently, and were replaced by two different kinds of successor.** Once people started replacing MapReduce with friendlier engines like Spark, Hive's SQL-to-MapReduce compiler simply became obsolete — that half of Hive had nothing left to compile *to*. What remained was Hive's table abstraction, and that half failed for reasons specific to a second migration: the move from local HDFS to cloud object storage. [[hive-object-storage-mismatch]] lays out four concrete failure modes — the Metastore's transactional guarantees don't extend to the underlying files, so Hive could only modify data at the whole-partition level; object storage rewards spreading reads across many prefixes while Hive's partition-per-prefix layout concentrates them; file listing before every read gets expensive at scale; and concurrent writers need Metastore locks that cap throughput. Tellingly, Hive's two problems were solved by two *different* successors: engines like Presto, Impala, and Drill replaced the query-compiler half by dropping MapReduce entirely, while Delta Lake, Iceberg, and Hudi replaced the table-abstraction half by re-engineering the metadata layer itself for object storage (see [[open-table-formats]]) — Hive's failure wasn't one event, it was two separate halves of the same system aging out on two separate timelines.

**Hive vs Presto as two generations of "SQL on the lake."** Where Hive compiles SQL down to a chain of MapReduce jobs and organizes state as HDFS folders behind a locking relational Metastore, [[presto]] was purpose-built in 2012 as a distributed SQL engine with no MapReduce dependency at all: a coordinator plans and orchestrates, workers execute against splits, and data moves between stages through in-memory buffered shuffles rather than being staged to disk between every step (see [[presto-coordinator-worker-scheduling]]). Presto also answers a problem Hive's papers don't really engage with — multi-tenancy and failure — with its own mechanisms: cooperative-multitasking CPU scheduling by accumulated CPU time, a two-tier user/system memory model with spilling and a reserved pool, and (for failures) external, not built-in, safety nets like standby coordinators and multiple active clusters (see [[presto-resource-management-and-fault-tolerance]]). And where Hive's optimizations are mostly about *where* data lives, Presto pushes optimization into *how compressed data is processed* — data skipping via file statistics, direct block conversion, and computing directly over dictionary/RLE-encoded blocks without fully decompressing them (see [[presto-columnar-execution-optimizations]]).

**Presto's own fork mirrors the pattern it emerged from.** Just as Hive's dominance didn't survive a governance and technology shift (MapReduce's replacement, the move to object storage), Presto's own governance shifted in 2018 when Facebook granted its own committers more commit privilege than the wider open-source community, prompting the original developers to leave and fork the project into PrestoDB (Facebook, later donated to the Linux Foundation) and PrestoSQL (the new Presto Software Foundation, rebranded Trino in December 2020 once Facebook held the "Presto" trademark). The same ecosystem that replaced Hive's compiler with Presto later saw Presto itself fork under pressure — a smaller-scale repeat of the pattern that runs through this whole history: no single system's dominance in data engineering has held for long before either the technology underneath it or the governance around it forced a fork or a replacement.

## Synthesis
Read end to end, this is a history of the same problem recurring at every layer: how do you let ordinary SQL users work with data that has outgrown the system it was first stored in? Codd's relational model answered it for a single machine; Inmon and Kimball each answered it differently for the enterprise warehouse, one centralizing first, one federating first. Hadoop's MapReduce answered it for web-scale raw data by throwing out the table entirely in favor of a low-level batch primitive, and Hive immediately patched that back in with a SQL compiler and a folder-based table abstraction over HDFS. But Hive's two halves — the compiler and the table abstraction — aged out on separate timelines and for separate reasons: the compiler became irrelevant once faster engines than MapReduce existed to compile *to*, and the table abstraction broke once the underlying files moved from HDFS to cloud object storage, whose flat, prefix-based, lock-free design Hive was never built for. Presto answered the first half by being a MapReduce-free SQL engine from day one, with its own scheduling, resource-management, and compressed-execution machinery; Delta Lake, Iceberg, and Hudi answered the second half by rebuilding the metadata layer itself for object storage. And even Presto wasn't immune to the same forces that unseated Hive — a governance dispute forked it into PrestoDB and Trino within a decade of its creation. The throughline isn't any one technology; it's that every generation's fix for "SQL over data that's outgrown its old home" becomes the next generation's thing to route around.
