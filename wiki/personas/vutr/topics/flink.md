## Comparisons
The spine of this topic is [[apache-flink]] itself: a distributed engine for "stateful computations over unbounded and bounded data streams" that treats everything as a stream, with batch as the special case — the opposite bet from Spark, which treats bounded data as first-class and aligns streams into micro-batches ([[batch-vs-stream-tradeoff]]). Flink's cluster model maps cleanly onto Spark's: JobManager ~ Driver, ResourceManager ~ Cluster Manager, TaskManagers ~ executors — but Flink foregrounds a resource-utilization/isolation trade Spark doesn't the same way, letting multiple tasks share a TaskManager's slots (multi-threaded, less isolated) or run one task per slot (isolated, less dense).

Internally, Flink avoids the JVM-object overhead that Spark's own Tungsten project was built to fight, via a custom serialization framework and fixed 32KB [[flink-memorysegments|MemorySegments]] allocated once per TaskManager and reused for its whole lifetime — a memory-management answer to the same GC-pressure problem Tungsten addresses on the Spark side. The same 32KB unit does double duty as Flink's network buffer, and [[flink-networking-and-flow-control|Credit-Based Flow Control]] is the mechanism that keeps that buffering from costing Flink its low-latency reputation: receivers grant senders credit, senders report backlog, and congested paths get more buffer space rather than throttling the whole job.

On fault tolerance, [[chandy-lamport-checkpointing]] is Flink's specific answer: a checkpoint barrier travels the dataflow graph, splitting it into pre/post-checkpoint state without ever pausing the application — in explicit contrast to the "pause, checkpoint, resume" approach both of Vu's Flink posts call out as impractical. [[flink-state-management-and-backends|State]] itself splits into Operator State (task-scoped) and Key State (per-key, co-located with matching records), backed by Java heap/off-heap memory, RocksDB, or — per a 2026 VLDB paper Vu cites — object storage directly.

On the streaming model, [[watermark]] and [[windowing-triggers-and-late-events|windowing/triggers]] are vocabulary Flink and the Dataflow paper share almost word for word: a watermark at T means the system assumes nothing with event time before T will still arrive, deliberately traded against accuracy (eager watermarks: low latency, more late-arrivals missed; relaxed: better accuracy, more waiting). Windowing (tumbling/sliding/session) manufactures the finite search scope that batch gets for free; a trigger decides when a window's result actually computes, independent of the window boundary itself. The [[dataflow-model|Dataflow model]] formalizes this further with [[dataflow-triggers-and-refinement-modes|AssignWindows/MergeWindows and three refinement modes]] (Discarding, Accumulating, Accumulating & Retracting) — a more general, engine-independent vocabulary for the same problem Flink solves concretely. Notably, the Dataflow paper frames its own triggers as a *single-pipeline* alternative to the Lambda Architecture's two-pipeline hedge: rather than a fast streaming estimate corrected later by a batch "truth" run, one pipeline emits multiple panes over time as its own understanding of completeness improves.

## Open questions
- **source gap**: Vu names "Spark Structured Streaming and Flink implement the watermark differently" but never spells out what that implementation difference actually is.
- **source gap**: the connection between [[chandy-lamport-checkpointing]] and Flink's exactly-once delivery guarantees is only pointed to via an external blog link, not explained — how does barrier-based checkpointing combine with sink-side behavior to actually deliver exactly-once?
- How does Flink's newer object-storage state backend (the 2026 VLDB paper Vu cites but doesn't detail) compare in practice to the RocksDB and heap/off-heap options — what's the latency/durability trade being made there?
- The Dataflow paper's refinement modes (Discarding, Accumulating, Accumulating & Retracting) are described at the model level — Vu's sources don't say which of these Flink itself exposes to users, or whether Flink's own windowing API even surfaces the AssignWindows/MergeWindows split explicitly.
- Vu flags that Flink's TaskManager slot-sharing is a resource-utilization/isolation trade but doesn't quantify it — how much utilization is actually gained, and at what measured cost to isolation, in a real cluster?

## Synthesis
Every mechanism in this topic traces back to the same fork: batch gets a bounded search scope and simple, replace-the-output idempotence for free ([[batch-vs-stream-tradeoff]]); streaming gives that up for latency, and every subsequent design choice — [[watermark|watermarks]], [[windowing-triggers-and-late-events|windows and triggers]], [[flink-state-management-and-backends|stateful operators]], [[chandy-lamport-checkpointing|non-pausing checkpoints]] — exists to buy back correctness and recoverability without paying batch's latency tax. [[apache-flink]] commits fully to the streaming side of that fork (everything is a stream, batch is the special case), and its internals — [[flink-memorysegments|MemorySegments]], [[flink-networking-and-flow-control|credit-based flow control]] — are engineering answers to the cost of that commitment: avoid JVM object overhead, keep buffering from becoming a latency tax. The [[dataflow-model|Dataflow model]] generalizes the same fork one level up, replacing "streaming vs batch" with "unbounded vs bounded" and refusing to assume completeness at all — a sharper, single-pipeline answer to the same problem Lambda Architecture solves by running two pipelines and reconciling them ([[dataflow-triggers-and-refinement-modes]]).
