## Comparisons
Three join strategies trade off along different axes, and the posts are explicit that all three reduce, physically, to a search problem. [[nested-loop-join]] needs no auxiliary structure at all and wins when the left table is small or the right table carries an index for the inner loop to exploit — its Block-Nested-Loop variant buys back some of the naive version's I/O cost by buffering a block of the left table rather than scanning row by row. [[sort-merge-join]] pays an up-front sort on both inputs but wins that cost back when the inputs are already sorted (or clustered-indexed) on the join columns, and it hands you sorted output for free if the query needs an ORDER BY on the join key. [[hash-join]] needs neither sorted input nor an index — it just needs the smaller ("build") table to fit in memory — and when it doesn't fit, it degrades gracefully into a Grace Hash Join that partitions both tables to disk and processes matching buckets one pair at a time, never bringing more than one bucket into memory. OLTP systems like PostgreSQL support all three, but OLAP systems narrow the field: with large tables on both sides and no point-lookup indexes, [[nested-loop-join]] loses both of its escape hatches, leaving [[sort-merge-join]] and [[hash-join]] to actually compete — and in practice hash join wins that competition at OLAP vendors: Snowflake uses it for most cases, BigQuery supports *only* hash joins, and only Spark offers sort-merge join as a real alternative. The distributed twist OLAP engines add on top of hash join is the broadcast hash join — shipping a small build table whole to every worker instead of shuffling both sides — which Spark lets you configure via a size threshold, BigQuery switches to dynamically (per the Dremel paper, even mid-query, by canceling an in-flight shuffle), and Snowflake detects automatically.

On the aggregation side, [[group-by]] and [[window-functions]] can compute the identical aggregate (say, `SUM(sales)` per product) but differ on exactly one point with large consequences: GROUP BY collapses the underlying rows into one summary row per group, while a window function returns a value for every original row without collapsing anything, because it's evaluated inside the SELECT stage of the [[sql-execution-order]] rather than as its own dedicated grouping stage beforehand.

On the algebra side, the operator taxonomy in [[selection-operator]] resolves a genuinely common confusion: Selection (σ) maps to WHERE, not SELECT — the operator that actually corresponds to SELECT is Projection (Π). Cartesian Product (×) is the operator every SQL JOIN is built from, with an equi-join being nothing more than a Cartesian Product followed by a Selection that keeps only rows where the join keys match — which is also exactly the logical-plan step the [[query-lifecycle]] performs before the optimizer picks a physical join algorithm.

Finally, [[materialization-strategies]] and [[cte]] answer a different question than any of the above — not *how* rows get combined or filtered, but *whether and when* a query's result gets physically stored. A table stores on every write; a materialized view stores once and goes stale until refreshed; a view never stores at all and re-executes every time; a CTE never stores anything and isn't even reusable outside the one query that names it.

## Open questions
- When the physical planner picks a join, is it choosing cost-based or rule-based in practice, and what statistics specifically tip it toward [[hash-join]] over [[sort-merge-join]] on a given OLAP engine?
- At what data volume does the Grace Hash Join fallback actually kick in for [[hash-join]], and how is the in-memory/spill threshold determined at runtime?
- Given [[oltp-vs-olap-access]], if a look-up index won't help OLAP the way it helps [[nested-loop-join]] in OLTP, which storage structures do minimize the volume of data read for a scan-and-summarize query instead?
- The posts assert Snowflake and BigQuery favor hash join over sort-merge join in OLAP but never explain *why* — is it implementation history, workload shape, or something structural about columnar storage that makes hash join preferable at that scale?
- Materialized-view refresh is described as scheduled, full, or incremental — but the posts don't say how a database decides which strategy to use for a given view, or what makes incremental refresh possible only sometimes.

## Synthesis
The through-line here is that SQL's declarative surface hides real, inspectable machinery underneath, and that machinery traces back to one historical move: Codd's separation of the logical relational model ([[relational-model]]) from physical storage, which is what makes a query language that says *what* rather than *how* possible in the first place. Relational algebra ([[selection-operator]]) is the layer that translates *what* into procedural steps — Selection maps to WHERE, Projection to SELECT, Cartesian Product underlies every JOIN — and the [[sql-execution-order]] governs how those steps actually run against real rows, which is why [[group-by]] collapses rows while [[window-functions]] don't. FROM needs somewhere to point, and [[materialization-strategies]] (table, view, materialized view) plus [[cte]] cover every shape that target can take, trading storage cost against freshness against reusability. Underneath all of it, the [[query-lifecycle]] parses, validates, plans, and executes every query, and the physical-plan stage is where [[nested-loop-join]], [[sort-merge-join]], and [[hash-join]] compete for the job — a competition [[oltp-vs-olap-access]] shows is workload-dependent, since OLAP's large-tables-both-sides, no-index reality eliminates nested-loop join outright and pushes real OLAP systems toward hash join, broadcast join, and Grace Hash Join's disk-partitioning trick when memory runs out.
