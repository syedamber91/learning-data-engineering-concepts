## Comparisons
**One substrate, several products.** [[procella]] (YouTube's query engine) and BigQuery's Dremel engine are presented in separate posts, years apart, but both rest on the same two Google-internal layers: [[colossus-and-borg-as-shared-substrate|Colossus for storage and Borg for compute]]. The posts don't dwell on this as a unifying theme, but naming Colossus and Borg underneath both systems is itself the clearest evidence in this material that "Google infrastructure" is one shared foundation wearing different product names — Procella, BigQuery, BigLake — on top, not three separate stacks that happen to share a vendor.

**Two answers to "how much should the query engine own the metadata."** [[biglake-tables]] and [[biglake-managed-tables]] both put a Google-built metadata layer (BigMetadata) in front of table data, but they disagree on how far to go. BigLake tables leave an open format like Iceberg in charge of its own metadata and use BigMetadata purely as a cache — file names, partitioning, per-file statistics — to cut down on repeated object-storage listing calls, while BLMTs hand BigMetadata full ownership of the metadata itself, replacing the open format's transaction log with a stateful in-memory-tail-plus-columnar-baseline design that trades open-format portability for higher mutation throughput and multi-table transactions the open formats don't support. The same delegated-access idea underlies both: neither trusts the query engine directly with raw object-storage credentials, because that would bypass BigQuery's own masking and row-level security ([[biglake-tables]]).

**Shared execution engines hiding under separate brand names.** [[superluminal-execution-engine|Superluminal]] was built for Procella — template-metaprogrammed, vectorized, works directly on encoded data, chosen specifically because Procella can't afford an LLVM compile pause on a millisecond-latency embedded-statistics query — and then resurfaces, per the BigLake post, inside the [[bigquery-storage-api|BigQuery Storage Read API]] and BigQuery BI Engine. The [[spark-bigquery-connector-vectorized-parquet-read|Spark BigQuery connector's Parquet fix]] is a concrete instance of this reuse working end to end: Google didn't build a new engine for Spark interop, it built a vectorized Parquet reader that feeds Superluminal's existing columnar batches directly, replacing an inefficient row-oriented round trip through Dremel's older Parquet reader.

**Data CI/CD as its own discipline, not a data-engineering afterthought.** The [[youtube-cicd-framework|YouTube CI/CD]] material treats data pipeline testing as fundamentally distinct from software CI/CD ([[data-pipeline-cicd-vs-software-cicd]]) rather than an application of the same practice to a different artifact: the reason a data test is hard to write isn't tooling gaps, it's that data's expected shape moves as the business does, which has no clean analogue in software testing's fixed expected outputs. The framework's answer — [[production-aligned-test-environments-via-config-rewriting|rewriting production configuration for an isolated subgraph]] rather than duplicating the whole environment — mirrors, at the pipeline-configuration level, the same instinct BigLake shows at the storage level: reuse the expensive, already-correct thing (production config; the open format's metadata) rather than rebuilding it from scratch.

## Open questions
- How does Colossus's high-latency, RPC-only metadata layer interact with Procella's registration server at YouTube's actual write volume — the post names the constraint (immutability, metadata server round trips, RPC-only access) but doesn't quantify how much registration-server-side batching or backoff is needed to make it tenable.
- **source gap** ([[procella-caching-and-affinity-scheduling]]): the post says affinity scheduling routes requests to a different server "when the desired server is down," but doesn't spell out how a request's original server is detected as down or how quickly the reroute happens.
- **source gap** ([[procella-tail-latency-mitigation]]): the post describes the RS asking "the secondary data server to come and be in charge of" a slow request, but doesn't explain how a secondary data server is selected, or what happens to the original (possibly still-running) request on the first server.
- Does BigLake's delegated-access connection object get revoked or rotated, and how does BigQuery detect that an underlying object-storage credential inside a shared connection has been compromised? The BigLake post describes the model but not its lifecycle management.
- The BLMT post claims higher mutation throughput than open table formats "without sacrificing read performance," but doesn't give a number — how much higher, and under what write concurrency does BigMetadata's in-memory tail itself become the bottleneck?
- The CI/CD post reports specific wins (99.9% data reduction, 50% faster integration investigation, schema changes in weeks not months) but doesn't say how those numbers were measured or over what baseline period — whether they're a single team's before/after or an aggregate across YouTube's warehouse.
- The **groupby-20** post in this group (`raw/google-infrastructure/groupby-20-how-google-takes-the-pain.md`) is a link-roundup newsletter issue, not original analysis — its one Google-relevant item is a pointer to an external Engineer's Codex article about Google's "Critique" code review tool, quoted only as a teaser of what that outside article covers. It contains no first-hand technical detail from vutr about Critique itself, so no concept or entity note in this group draws on it; it's copied to `raw/` for provenance but otherwise contributes nothing groundable.

## Synthesis
Read together, these four posts describe a single company's infrastructure showing up from three different vantage points: a purpose-built serving/analytics engine (Procella, at YouTube's product-team scale), a general-purpose warehouse product extending itself toward the lakehouse (BigQuery, via the [[bigquery-storage-api|Storage API]] and [[biglake-tables|BigLake]]/[[biglake-managed-tables|BLMT]]), and the operational discipline needed to ship changes safely across thousands of pipelines built on top of all of it (the [[youtube-cicd-framework|CI/CD framework]]). What ties them together isn't branding — it's [[colossus-and-borg-as-shared-substrate|Colossus and Borg underneath both engines]], [[superluminal-execution-engine|Superluminal underneath both execution paths]], and a recurring design instinct to make the expensive, already-correct thing reusable — production config in CI/CD, an open format's metadata in BigLake, a cached file handle in Procella — rather than rebuild it per use case. The trade each layer makes for that reuse is visible too: Procella accepts caching complexity and affinity-routing subtlety in exchange for treating Borg's instability as normal; BigLake tables accept a cache-only relationship with open-format metadata to stay portable, while BLMT gives up that portability for throughput and audit guarantees the open formats don't offer; and YouTube's CI/CD framework accepts the cost of a Configuration Rewriter and a dependency-graph-aware test harness in exchange for not having to duplicate its production environment for every test.
