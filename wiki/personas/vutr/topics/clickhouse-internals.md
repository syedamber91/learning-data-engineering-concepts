## Comparisons
**Self-managed MergeTree vs Tinybird's object-storage model.** Both ultimately run ClickHouse, but they take opposite bets on where state lives. A self-managed cluster keeps every replica's parts on local disk, coordinated by [[clickhouse-keeper]]'s Raft-based multi-master scheme ([[clickhouse-replication-and-keeper-consensus]]) — durable and fast to read locally, but the operator owns cluster sizing, sharding strategy, Keeper-vs-ZooKeeper choice, and rebalancing whenever membership changes. [[tinybird]] inverts this: replicas hold data in object storage and use local SSDs only as cache, which makes them stateless and lets the platform scale horizontally with no data-rebalance step ([[tinybird-ingestion-and-scaling]]). Vu never states that Tinybird's replicas skip ClickHouse's own replication log entirely, but the object-storage move clearly targets the same operational pain — cluster coordination and data movement — that [[clickhouse-replication-and-keeper-consensus]] describes as the cost of self-management.

**Mutation strategy is a genuine three-way choice, not one mechanism.** MergeTree offers a full rewrite mutation (correct but expensive — an entire part rewritten for even a single-row change), lightweight deletes (a bitmap flip, fast to write but slightly slower to read since every scan checks the bitmap), and — for workloads with heavy mutation volume specifically — a dedicated engine, ReplacingMergeTree, that reframes "mutate" as "insert a newer version and let merge discard the old one" ([[mergetree-merge-and-mutation]]). ClickHouse's own internal data warehouse picks the third option for a completely different reason than mutation-heaviness: ReplicatedReplacingMergeTree is their answer to Airflow DAG retries re-sending the same hour of data, i.e. it's their idempotency mechanism, not their update mechanism ([[clickhouse-internal-data-warehouse-case-study]]).

**Two different consistency knobs solve two different problems.** `insert_quorum` (see [[clickhouse-insert-process-and-idempotency]]) is a write-time setting: it makes an insert wait for full replication before reporting success, closing the specific gap where a downstream read sees a partially-replicated write. The hash-table dedup behind idempotent insert is a completely separate mechanism solving a client-visibility problem (did my insert actually land after a timeout), not a replication-visibility problem. ClickHouse's own warehouse needs both simultaneously, for two unrelated failure modes in the same Airflow-orchestrated pipeline.

**Same product, opposite relationships to operational burden.** [[clickhouse-internal-data-warehouse-case-study]] is ClickHouse eating its own cooking — Airflow, S3, dbt, and Superset wired directly against self-managed ClickHouse Cloud, with the team owning every operational decision described in [[clickhouse-keeper]] and [[clickhouse-replication-and-keeper-consensus]]. [[tinybird]] exists to let a *different* company skip owning any of that: managed infrastructure, ingestion flow control with a Kafka-style dead-letter-queue pattern for bad data ([[tinybird-ingestion-and-scaling]]), and endpoint pipes that remove the need to hand-build a backend server at all ([[tinybird-pipes-serving-and-dev-experience]]). Same underlying engine, two case studies that sit at opposite ends of how much of ClickHouse's operational surface a team chooses to own.

## Open questions
- The source names "opportunistic code compilation" as part of ClickHouse's query engine ([[clickhouse-vectorized-parallel-execution]]) but never explains what triggers it, what gets compiled, or how it interacts with the vectorized execution path running alongside it.
- The background merge process is said to pick parts using "heuristics, such as part sizes, currently executed merges, or the number of parts" ([[mergetree-merge-and-mutation]]) — but the actual selection algorithm or thresholds are never spelled out.
- The idempotent-insert hash table tracks "recently inserted parts" ([[clickhouse-insert-process-and-idempotency]]) — what exactly keys an entry (content hash? block number? a client-supplied ID?), and what evicts an entry from that table over time?
- ClickHouse Keeper's Raft consensus is named as backing the replication log ([[clickhouse-replication-and-keeper-consensus]]), but the source never explains how concurrent writes from different nodes get serialized into that log before replay — is there a Raft leader for the log itself, or does "multi-master" mean something else at that layer?
- Tinybird's ingestion flush interval "depends on the billing plan" ([[tinybird-ingestion-and-scaling]]) but no actual numbers are given — there's no way to know from the source how much latency the batching trade-off actually costs at any tier.
- **source gap** ([[sparse-index-and-read-path]]): the source describes table projections as "lazily facilitated only from parts newly inserted... unless the user fully materializes the projection," but doesn't explain what triggers or how the system executes that full materialization.
- **source gap** ([[clickhouse-internal-data-warehouse-case-study]]): the source says dbt was adopted "to centralize the transformation logic," but doesn't describe what concretely changed in the Airflow DAG structure or staging-table pattern once dbt took over — did Airflow's single main transformation DAG shrink, get replaced, or keep orchestrating dbt runs?
- **source gap** ([[tinybird-ingestion-and-scaling]]): the source says replicas cache object-storage data on local SSDs, but never addresses cache coherence — how a replica's local cache is invalidated or kept consistent once the underlying object-storage data changes.

## Synthesis
Every layer of ClickHouse's design makes the same trade in a different place: never let a per-row cost sit on the synchronous write path. The sparse index defers per-row bookkeeping to a per-granule mark ([[sparse-index-and-read-path]]); the MergeTree part model defers consolidation to a background merge thread instead of writing in place ([[mergetree-storage-engine]], [[mergetree-merge-and-mutation]]); mutation itself is asynchronous and batched by default; async insert mode defers part creation until a buffer fills; and replication defers convergence to asynchronous log replay under eventual consistency, with `insert_quorum` available as the deliberate escape hatch when a workload can't tolerate that default ([[clickhouse-replication-and-keeper-consensus]], [[clickhouse-insert-process-and-idempotency]]). ClickHouse's own internal data warehouse shows this philosophy scaled up to a whole company's data platform — hourly incremental loads, staging tables per DAG run, ReplicatedReplacingMergeTree absorbing Airflow's retries — and Tinybird shows it scaled up again, deferring not just per-row cost but the *entire operational surface* of running ClickHouse to a managed platform, so a team pays in ingestion latency what it would otherwise pay in months of cluster-tuning work. The through-line across four articles: ClickHouse's speed and its operational shape are the same decision, viewed at different altitudes.
