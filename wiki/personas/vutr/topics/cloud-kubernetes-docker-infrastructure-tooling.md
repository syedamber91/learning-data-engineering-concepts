## Comparisons
**One spectrum, four stops: VM → container → orchestrated cluster → serverless.** [[cloud-compute-abstraction-spectrum]] is Vu's own abstract claim — every compute service is "backed by the same physical servers," and what changes is how much of that server you're forced to see. [[docker]] and [[kubernetes]] are what actually fill in the middle two stops he only gestures at. Docker answers "how do I stop needing a full guest OS per app on one machine" by sharing the host kernel across containers; Kubernetes answers the next question Docker's own architecture can't — "what happens when the containers outgrow one machine" — via a controller that continuously reconciles a declared desired state against the cluster's observed actual state. Vu is explicit that comparing the two feature-for-feature is the wrong instinct for a newcomer; they solve different-order problems, and his own onboarding mistake with Docker was skipping straight to commands without first asking which problem the tool solves at all. [[aws-glue]] then shows what the *top* of that same spectrum — true serverless — costs to build: Glue's three engine generations are a direct, measured record of trying to make the underlying compute invisible, from Glue 1.0's 8–10 minute cold-start cluster provisioning down to Glue 2.0's sub-10-second cold starts and Glue 3.0's mid-job auto-scaling — the VM is never gone, but each generation makes it recede further from the user's view.

**Kubernetes' own reconciliation model recurs one layer down, inside Kubernetes itself.** The cluster-level "desired vs. actual state" loop that defines Kubernetes as a whole ([[kubernetes]]) is the same mechanism [[kubernetes-workload-abstractions|Deployments]] use to keep a Pod count steady, and it's also exactly what the [[spark-on-kubernetes-execution-model|Spark Operator]] adds on top when Spark applications didn't fit that model natively: a Custom Resource Definition plus a custom controller that reconciles a `SparkApplication` object the same way a built-in controller reconciles a Deployment. Three different workloads (a generic app, a stateful database, a transient Spark driver/executor pair), one reconciliation idea, reused rather than reinvented at each layer.

**Deployment vs. StatefulSet is the concrete version of "identity you don't need" vs. "identity you can't lose."** [[kubernetes-workload-abstractions]] draws this out with Vu's own worked contrast: a Deployment treats every Pod as interchangeable, which is exactly wrong for his PostgreSQL primary-replica example — a recreated Pod there would get a new name, empty storage, and no ordering guarantee, which between them would break replication three separate ways. StatefulSet exists purely to remove those three failure modes (stable naming, per-Pod PersistentVolumeClaims, ordinal-ordered scaling) for the specific case where a Deployment's interchangeability is the bug, not the feature.

**Two different orchestration answers to "stitch several AWS services into one pipeline," never actually compared against each other in the sources.** [[aws-event-driven-fanout-architecture]] (a guest post on Vu's newsletter) hand-wires S3, SNS, SQS, Lambda, and Step Functions into an event-driven fan-out, explicitly weighing that pattern against a centralized modern orchestrator like Airflow and landing on "it depends" — new builds favor Airflow, existing constrained environments favor bolting one more native service on. [[aws-glue]] solves a narrower version of the same stitching problem with its own **workflows** layer, chaining Crawlers and Glue jobs with parameter passing and schedule/event triggers — but scoped entirely inside Glue's own ETL world rather than across arbitrary AWS services the way Step Functions can. Both are orchestration mechanisms living natively on AWS; neither source discusses when you'd reach for one over the other.

**Cost and access fundamentals are the lens applied to every specific service above them, not a separate topic.** [[cloud-access-cost-and-observability-fundamentals]] and [[cloud-regions-and-availability-zones]] read as abstract advice in isolation, but every other concept in this group is a concrete instance of that lens: a Kubernetes cluster's nodes still incur cross-AZ networking cost exactly as [[cloud-regions-and-availability-zones]] describes for any other service; Glue's own warm-pool and auto-scaling generations exist specifically because uncontrolled compute cost and unpredictable start times were the problem being solved; and Vu's IAM "just enough permission" principle (a pipeline that reads S3 shouldn't delete databases) is the same instinct behind [[kubernetes-workload-abstractions|Kubernetes RBAC]] scoping a service account to the `edit` clusterrole rather than cluster-admin for a Spark submission.

**Object storage is the physical floor everything else in this group sits on.** [[object-storage-as-data-lake-backbone]] names S3 "the backbone of every modern data architecture" and the correct first cloud service to touch — and every other concept here bears that out concretely rather than abstractly: [[aws-glue]]'s entire reason for existing is discovering, cataloging, and transforming data that lands in S3 first; [[aws-event-driven-fanout-architecture]]'s whole pipeline begins with an S3 event; and Vu's own on-ramp advice (learn storage classes, lifecycle rules, and LIST/PUT/GET before touching compute at all) is the same ordering he applies when introducing Docker and Kubernetes only after object storage and VMs are already comfortable.

## Open questions
- **source gap** ([[kubernetes]]): the source names etcd as the key-value store holding cluster state and calls it "consistent" but doesn't spell out the Raft-based replication/election mechanics that actually provide that consistency guarantee.
- **source gap** ([[kubernetes]]): the source describes the Scheduler as assigning Pods to Nodes "based on resource availability and policy" but doesn't spell out the actual scoring/filtering algorithm behind that assignment.
- **source gap** ([[docker-image-container-and-runtime]]): the source names the OCI bundle as what `containerd` hands to `runc` but doesn't spell out what that bundle actually contains, or how `runc` uses kernel primitives (namespaces, cgroups) to build isolation once it talks to the OS.
- **source gap** ([[glue-serverless-execution-evolution]]): the source says Glue 2.0's resource manager uses "ML models to predict EC2 instance demand across regions" but doesn't spell out what those models are or how their predictions are validated.
- **source gap** ([[spark-on-kubernetes-execution-model]]): the source names the Spark Operator's custom controller as reconciling a `SparkApplication` CRD but doesn't spell out how it derives concrete driver/executor Pod specs (resources, scheduling constraints) from that declarative object.
- Vu's "VMs, one serverless option, and to some extent Kubernetes are enough" claim ([[cloud-compute-abstraction-spectrum]]) is framed explicitly as onboarding advice for someone starting from zero — the source never revisits whether that minimal set still holds once IaC, multi-region deployment, and full observability get layered on for an actual production data platform.
- [[aws-event-driven-fanout-architecture]] and [[aws-glue]]'s own workflows both solve "stitch AWS services into a pipeline," but the sources never compare them directly — when would Step Functions' cross-service reach beat Glue workflows' narrower-but-native ETL scope, or vice versa? Left unaddressed.

## Synthesis
Vu's cloud-and-infrastructure material builds one stack from the ground up. Underneath every specific service sits a fundamentals layer — region/AZ choice, IAM and networking, cost control, observability — that he insists gets learned once and reapplied everywhere rather than relearned per service ([[cloud-access-cost-and-observability-fundamentals]], [[cloud-regions-and-availability-zones]]). Object storage is the first concrete service worth touching, because it's cheap, safe by default, and the literal backbone every data lake and pipeline in this group lands data into first ([[object-storage-as-data-lake-backbone]]). Compute above that is a spectrum of how much of the underlying VM you're forced to see, from renting one directly to a managed Kubernetes service to true serverless ([[cloud-compute-abstraction-spectrum]]) — and Docker and Kubernetes are what make the middle of that spectrum operable at all: Docker solves single-host packaging and isolation ([[docker]], [[docker-image-container-and-runtime]]), Kubernetes solves multi-host scheduling and self-healing via a desired/actual-state reconciliation loop that recurs at every layer built on top of it, from a plain Deployment to the Spark Operator's own CRD-driven controller ([[kubernetes]], [[kubernetes-workload-abstractions]], [[spark-on-kubernetes-execution-model]]). AWS Glue is Vu's own worked example of pushing one specific workload — data integration ETL — as far toward that serverless end as it will go, its three engine generations a measured record of shrinking cold-start time and infrastructure awareness down toward zero, backed by its own metadata layer (the Glue Data Catalog and its crawlers) built to replace the operational burden of a self-managed Hive Metastore ([[aws-glue]], [[glue-serverless-execution-evolution]], [[glue-data-catalog-and-crawlers]], [[glue-dynamicframe-and-schema-inference]]). The guest-post fan-out architecture ([[aws-event-driven-fanout-architecture]]) closes the loop by showing the same primitives — object storage, pub-sub, serverless functions, an orchestrator — composed by hand rather than through a managed data-integration service, an alternative Vu candidly weighs against reaching for Airflow instead.
