Storage has been getting cheaper, and SQL OLAP systems have become more powerful.

This helps move from ETL to ELT.

The transformation moves from outside to inside.

SQL, which already played an important role in serving data, has become crucial as a dominant option for transforming data.

Data transformation has been democratizing.

The transformation logic, handled by engineers with robust coding skills, can now be written by a data analyst or an analytic engineer, with SQL queries.

However, the logic still needs to be tested, modularized, and documented.

For SQL queries, we need a tool to help us do all of that.

That’s why dbt was created.

It is a CLI tool that lets us efficiently transform data with SQL. Organizing data transformation into models, automating doc generation, and letting developers define tests.

A dbt model is purely Jinja + SQL code. Thus, it can be tracked for changes, rolled back to previous versions, and implemented with CI/CD pipelines.

Just like software engineers do with application code.
