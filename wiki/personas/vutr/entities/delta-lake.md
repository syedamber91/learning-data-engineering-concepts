Delta Lake's founding problem, per Databricks' own 2020 paper (which the notes work through directly), is that cloud object stores are key-value stores: they give you per-object durability and (eventually) per-key consistency, but no atomic transaction across multiple objects, and expensive, high-latency LIST operations (S3 returns at most 1,000 keys per call, tens to hundreds of milliseconds each). Treat a table naively as "a set of Parquet objects" and you get no cross-query isolation, a corrupted table on any mid-update failure, and slow metadata operations at scale. Delta Lake's answer, stated plainly in the notes: keep information about which objects belong to a table in an ACID manner, using a write-ahead log stored in the object store itself, next to Parquet data objects that any Parquet-aware engine can already read.

**The log.** A Delta table is an object-storage directory holding Parquet data objects plus a `_delta_log` subdirectory: a sequence of JSON files with zero-padded, monotonically increasing IDs (`000001.json`, `000002.json`, ...). Every data file in the table must be referenced from this log — a file no log entry points to is not readable. Each log record is a set of actions applied to the previous version to produce the next: `metaData` (schema, partition columns, format, config — first version must have one, later ones fully replace it), `add`/`remove` (which objects are now in the table, with `add` optionally carrying per-column min/max statistics, and `remove` carrying a deletion timestamp so the underlying object survives until a retention threshold passes), `protocol` (bumps the Delta protocol version), `commitInfo` (provenance), and `txn` (lets an application track its own progress inside the log — e.g., a streaming job recording which source offset it last wrote, so a restart resumes from the right place rather than replaying or dropping data). Periodically the log is compacted into a Parquet checkpoint holding only the non-redundant actions up to some log ID (superseded `add`s dropped, `remove` tombstones kept per retention policy), with the latest checkpoint's ID recorded in `_delta_log/_last_checkpoint` so clients never have to scan the whole log directory just to find where to start.

**Read and write protocols.** Reading: check `_last_checkpoint`, LIST for any newer JSON/checkpoint files past that point, replay them to reconstruct the table's current set of objects and statistics, prune with those statistics, then read the surviving objects (in parallel, retrying on any object a worker can't yet see because of the store's eventual consistency). Writing: identify the current log version, read that state, write new Parquet objects in parallel, then attempt to commit the *next* sequential log ID — and only if nobody else already claimed that name.

That commit step is Delta's whole concurrency story, and it is the concrete case covered in [[occ-on-object-storage]]: it is [[conditional-writes|conditional writes]] (put-if-absent) end to end. Google Cloud Storage and Azure Blob Storage offer put-if-absent natively; on HDFS, Delta uses atomic renames instead. Amazon S3 didn't support put-if-absent until August 2024, so pre-2024 Delta-on-S3 needed an external DynamoDB-backed lock service just to serialize who could claim the next log ID — a dependency the notes expect Databricks no longer needs now that S3 supports native conditional writes. The notes walk a concrete two-writer race: both start from `000001.json`; writer B wins the put-if-absent for `000002.json` first; writer A's own attempt to write `000002.json` fails outright (an unconditional retry would have silently orphaned B's file), so A must reload the table, check for conflicts against B's committed change, and — if clear — retry as `000003.json`. All Delta write transactions are therefore serializable by construction (one winner per log ID, in ID order); reads default to snapshot isolation (a fixed log version) and can force serializability with a dummy write. Because that whole path rides on put-if-absent latency (tens to hundreds of milliseconds per call), Delta's write-transaction rate tops out at "several transactions per second" — the same ceiling this vault's [[occ-on-object-storage]] note flags as an open comparison point against Iceberg's and Hudi's own commit throughput.

**Data mutation.** Delta originally supported only copy-on-write: any UPDATE/DELETE rewrites the whole affected Parquet file(s), so deleting one row from a file with millions still means writing a new file with all the others. Delta Lake later added **deletion vectors (DVs)** — small files that mark rows in a data file as logically deleted without rewriting it, functioning as Delta's answer to merge-on-read (see [[copy-on-write-vs-merge-on-read]] for how this compares to Iceberg's positional/equality delete files and Hudi's log files). The notes trace a concrete example: inserting two rows writes File 1 (log: `add File 1`); deleting one row writes DV file 1 (log: `add File 1 + DV 1`, `remove File 1` to signal the mutation); updating the other row's value writes a new File 2 plus a new DV file 2 that folds in the prior deletion location (log: `add DV 2 + File 2`, `remove DV 1`) — so a client reading right after must merge three files (File 1, DV 2, File 2) to reconstruct the table.

**Beyond the core protocol**, the notes cover several higher-level features built on the same immutable-log foundation: time travel and rollback (query an older log version directly, or `CLONE` a specific version); the log doubling as a message queue for streaming (write compaction of small streamed objects, `txn`-based exactly-once writes, and cheap "log tailing" thanks to the log's lexicographically increasing JSON filenames); and layout optimization (`OPTIMIZE` to compact objects toward a target size, `AUTO OPTIMIZE`, and Z-ordering — reorganizing records so each object covers a narrow range of chosen attributes, which the notes report skips at least 43% of Parquet objects for single-dimension queries and 54% on average across dimensions). Caching is safe specifically because Delta's data and log objects are immutable, and audit logging rides on the log's own `commitInfo` records.

Delta Lake is more Spark-centric than Iceberg or Hudi — a difference DoorDash (see [[doordash-iceberg-realtime-migration]]) cites as a concrete reason it didn't standardize on Delta for its Flink-heavy workload. Walmart's own benchmark (see [[walmart-hudi-benchmark]]) doesn't make this Spark-centricity argument at all — Delta actually outperformed in most of Walmart's query benchmarks, and Walmart chose Hudi instead for breadth of batch-plus-streaming support and upsert/delete efficiency, not because Delta was too Spark-centric.
