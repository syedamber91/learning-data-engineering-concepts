Dremel is BigQuery's distributed query engine — the component that turns a submitted SQL string into a running, adaptive execution graph. Google's own architecture sketch names four independent pieces around it: Colossus for storage, Borg for cluster management, Dremel itself for query processing, and a dedicated shuffle service built on disaggregated distributed memory that also backs other products like Dataflow. Submitting a query is itself a small distributed journey: the request lands on a Google Front-End (GFE) server (the same tier that serves Google Search), gets routed to the BigQuery Job Server — which tracks request state asynchronously because the client connection can drop and a query can run for hours, and also handles authorization — and the Job Server hands off to the Query Master (also called the Query Coordinator).

The Query Master's job is planning: parse the query to find which tables and filters are involved (filters double as partition-pruning hints), look up table metadata from Cloud Spanner (skipping files outside the matched partitions), and build a query plan divided into stages, each stage a set of operators run in parallel across workers. Dremel's plan shape has itself evolved. Originally (Google's paper era) the execution plan was a **static** multi-level tree with a fixed number of workers decided up front: a root node takes the query, rewrites and routes it downward, leaf nodes touch storage, and results/aggregations flow back up through the intermediate levels. That fixed-tree design worked for scan-filter-aggregate queries but not for joins or nested subqueries, and Dremel also runs "in-situ" over previously unseen data, so accurate statistics for planning aren't available up front. In 2010 Google moved to a **dynamic** plan: the number of workers per stage, the partitioning of intermediate data, and even the join strategy can all change mid-query based on statistics collected during execution.

Once the Query Master has an initial plan, it asks the Scheduler for slots — Dremel switched to *centralized* scheduling in 2012, using whole-cluster state to make finer-grained allocation decisions than a distributed scheme could, which improves both utilization and isolation. A slot corresponds to one execution thread on a leaf server (3,000 leaf servers at 8 threads each is 24,000 slots), and the Scheduler shares slots fairly among equal-priority users but can preempt a lower-priority query's running slots to make room for a higher-priority one. The actual work happens in **Worker Shards** — tasks running inside Borg (Kubernetes's ancestor) so Dremel doesn't have to manage infrastructure directly. A Worker Shard runs multiple threads (slots) in parallel; each slot typically owns one input file, reading either from Colossus or from the output of a prior stage sitting in the shuffle layer, and writing its own output to that same shuffle layer (spilling to disk if it doesn't fit in memory). Every unit of work is atomic and idempotent, so a failed job can simply be restarted without side effects.

The serverless design rests on three ideas Google states directly: separating compute, storage, and memory so each can scale independently; assuming underlying compute can be slow or fail, which is why every sub-task must be deterministic and repeatable (only the failed slice needs restarting) and why multiple copies of the same task can be scheduled speculatively across workers, with the first result to finish simply kept; and scheduling in abstract units of compute/memory called **slots** rather than concrete machine shapes — slots remain BigQuery's customer-visible resource concept today.

Because the shuffle layer holds all intermediate state (see [[disaggregated-shuffle-service]]), the Query Master can treat a shuffle's output as a checkpoint of query-execution state and dynamically resize the next stage's worker count — this is also what makes the join strategy adaptive. Dremel supports both **broadcast joins** (a small table is sent whole to every slot processing the large table) and **hash joins** (both sides are shuffled so matching keys land in the same slot — expensive because data has to move). Dremel starts a join by shuffling both sides for a hash join, but if one side finishes fast and its size falls under a broadcast threshold, it cancels the second shuffle mid-flight and switches to a broadcast join instead. The same shuffle-driven visibility lets Dremel detect **data skew** — one shuffle partition getting overloaded relative to others — and react by instructing workers to do another round of hashing, routing the overloaded partition's data into new partitions consumed by a freshly brought-up repartition worker, after which the old, overloaded partition is discarded.
