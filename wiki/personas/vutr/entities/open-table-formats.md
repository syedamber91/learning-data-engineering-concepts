Object storage only ever gives you objects — a table format's whole job is to make a query engine able to look at a pile of Parquet files and "see" a table. Every database (Snowflake, BigQuery, Redshift, Databricks, even Postgres) needs some translator layer to do this; what makes Iceberg, Delta Lake, and Hudi *open* table formats specifically is that this translator — the metadata layer — is separate from any database, has no database dependence, and lives alongside the data on object storage itself. That is the whole reason "open" comes before "table format."

The metadata layer records everything a table needs beyond raw bytes: table name, schema, column types, snapshots, file locations, partition scheme, column statistics, and more. Each format names and structures this differently — Iceberg's manifest files and metadata files organized as a tree (see [[iceberg-metadata-layer]]), Delta Lake's transaction log, Hudi's Timeline (see [[hudi-timeline]]) — but underneath the naming they solve the identical problem: how does a query engine work on a pile of files as if it were working with a table? That shared problem produces a shared read/write pattern across all three formats: **write** always puts data files down first and metadata files after; **read** always goes to metadata first to learn which data files to open. Schema evolution rides on the same separation — because schema lives in metadata rather than being baked into the physical files, the metadata layer can track every version of the schema a table has ever had and exactly when each version took effect, so old and new files can coexist under different historical schemas and a reader simply applies whichever schema was in force when a given file was written.

**Where this need came from.** The historical arc runs from data warehouse (schema-on-write, centralizing structured reporting data from a small number of sources) to data lake (Hadoop/HDFS, later object storage, storing data in native format with no upfront schema) to lakehouse. Warehouses hit a wall as unstructured and semi-structured data multiplied; lakes solved storage flexibility but, taken alone, lack the table abstraction, ACID guarantees, and DML support that make a warehouse usable — "data is just a bunch of files." Meta's 2009 Hive paper was an early attempt to bring table abstraction to Hadoop, and Google's Dremel/"in-situ" processing vision (later public as BigQuery) chased the same goal from a different angle, but both stayed largely big-company efforts. Databricks named "lakehouse" in 2020, framing the two-tier warehouse-plus-lake status quo's real costs plainly: keeping lake and warehouse consistent is expensive, the warehouse copy is always stale relative to the lake, advanced analytics support is limited, and running both means double storage plus ETL cost.

Hive itself is the direct reason the next generation of formats had to exist. Its four concrete failure modes, per the notes: (1) a transaction mismatch between the Hive Metastore (a transactional relational database) and the underlying filesystem (which guarantees nothing), so a write interrupted mid-flight can leave corrupted data; (2) concurrent writers had to take out Metastore-level locks, which throttles throughput; (3) expensive file listing before every read, since the engine must enumerate a partition's files before it knows what to open; and (4) that listing problem gets structurally worse on object storage specifically, because object stores reward spreading reads across many distinct key prefixes for parallelism, while Hive's partition-as-prefix convention concentrates reads on the same prefix. On top of all that, Hive originally only supported mutating data at the whole-partition level — exactly the constraint that forced Uber into repeatedly recopying entire partitions for single-row changes and drove it to build Hudi (see [[apache-hudi]]).

**What the format is actually responsible for.** Of ACID's four properties, Consistency and Durability arrive nearly free: Durability comes from the object store itself (S3 and GCS both advertise roughly eleven-nines durability), and Consistency is mostly a client-side contract the format can only partially help enforce (e.g., not-null checks) since real business-rule validation happens in the application. That leaves Atomicity and Isolation as the format's actual engineering problem — see [[occ-on-object-storage]] for how each of the three formats solves it. Physical-layout pruning is the other pillar of format design, working at three levels — partitioning (splitting the table so an engine touches only relevant portions), clustering/sorting (collocating related rows within files), and per-file column statistics (letting an engine skip whole files based on recorded min/max/null counts) — and because every table format can only ever append new immutable files, compaction of small files into fewer, larger ones is not optional tuning but a structural necessity every one of these formats has to build in.
