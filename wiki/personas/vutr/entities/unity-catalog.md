Unity Catalog (UC) is Databricks' central metadata and governance service for the lakehouse — proprietary for years before Databricks open-sourced it. Per the paper the notes work through, it was already serving roughly 9,000 Databricks customers with 100 million tables under management, 400,000 ML models, and 60,000 API calls per second before that release. A catalog, in the general database sense the notes open with, is a central repository of metadata (tables, columns, views, users, relationships) that lets a system find and manage data; UC's specific twist is doing that job across a genuinely lakehouse-shaped stack — object storage underneath, Delta Lake or Iceberg providing the table abstraction on top of it, and now ML models and other asset types alongside tables.

**The problems it names as its reason to exist:** uniform access control regardless of whether a user reaches an asset via the catalog's own API or a direct cloud-storage path; support for asset types beyond tables (starting with ML models); external data sharing without copying data (avoiding both the storage cost and the synchronization burden of a second copy); discovery — assets need to be findable, with transparent lifecycle and lineage; and raw performance, since UC sits in the hot path of every query even though it only touches metadata.

**Where it sits.** The Databricks lakehouse stack the notes describe has three parts: the storage layer (object storage plus a Delta Lake or Iceberg metadata layer providing the table abstraction — see [[apache-iceberg]] and [[delta-lake]]); the Databricks Runtime (DBR), a Spark fork enhanced with the Photon vectorized engine as an additional set of physical operators inside Spark's own query plan, so existing workloads benefit without code changes; and the Unity Catalog service itself, a multi-tenant service exposing UC's functionality via APIs that query engines call into. A typical query's path: DBR parses the SQL to find the referenced tables/views, sends a REST request to UC to check permissions and fetch metadata (schema, constraints), plans using that metadata, requests a short-lived credential from UC when it actually needs to touch object storage, and only then executes.

**Design choices, and the problems each one answers:**

- *Disaggregating the catalog from the engine.* UC is an independent service, not tied to any one query engine — which buys both security (UC is the sole gatekeeper any engine has to go through) and interoperability (different engines call the same REST API instead of each replicating a private copy of the metadata).
- *Diverse asset types.* The foundation is an entity-relationship model, materialized in a relational database, that exposes generic operations — ID/name lookup, parent-child relationships, privilege grants, lifecycle/state management. An adapter layer sits on top to integrate specific asset types with specific cloud providers, and a developer registers a new asset type by adding a manifest describing its place in the model hierarchy, its supported operations/privileges, and its lifecycle rules — which is how ML models were added alongside tables without redesigning the core.
- *Access control.* Databricks protects assets at two levels. Metadata access is checked directly against UC's own REST APIs per operation. Data access is stricter still: rather than grant clients direct cloud-storage permissions at all, administrators grant storage permissions *only* to UC itself — any client, even one accessing a table "by path" rather than by catalog name, must first have UC resolve that path back to an asset, check permissions, and hand back a short-lived credential. On top of that metadata/data split, the query engine itself enforces a second, finer-grained layer (row- and column-level security).
- *Discoverability.* Search, lineage, and other background functions sit above the core service and stay updated via an event-driven model — the core emits change events whenever metadata changes, and background functions consume them to update their own indexes rather than being queried synchronously. Keeping background functions decoupled from the core service this way means either can scale, or fail, independently.
- *Performance.* UC batches multiple asset requests together, caches immutable metadata (like temporary credentials) at both the UC service and the calling engines, and — for mutable metadata such as table names or permissions — uses a write-through cache backed by a relational database sharded across nodes, so each node's cache and storage stay co-located for its assigned data subset. To bound cache growth, UC combines standard LRU eviction with a hard per-call timeout: once a request populates a new cached version of an asset, existing cached copies are only trusted for at most that timeout window, which caps how many stale versions of a popular entity can accumulate. For its own metadata database, UC uses database versions to provide snapshot isolation and serializability — the identical isolation-level ladder this vault's notes describe generally (see [[isolation-levels-and-mvcc]]) and that the table formats themselves reuse for their own optimistic concurrency control (see [[occ-on-object-storage]]).

Because UC's own "storage" layer is explicitly Delta Lake or Iceberg tables, it functions as a governance layer that sits *above* either format's own catalog concept — a contrast worth holding next to Iceberg's own lighter-weight REST catalog specification (the one Netflix's Polaris implements; see [[netflix-iceberg-ecosystem-and-migration]]), which standardizes table-metadata access without taking on UC's broader asset-type, sharing, and lineage scope.
