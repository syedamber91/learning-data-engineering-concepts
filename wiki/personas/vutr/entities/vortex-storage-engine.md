Vortex is the storage engine Google built underneath BigQuery to support real-time analytics. Its founding observation, per the paper, is that adapting infrastructure built for batch data to also handle streaming is the wrong direction — better to build for streaming first and extend that system to batch. Vortex provides a highly distributed, synchronously replicated engine optimized for append-heavy ingestion, which is why Vu compares its two-format design to ClickHouse's MergeTree engine.

The core abstraction is the **Stream**: an entity clients append rows to, where every row is identified by its stream ID plus an offset. A table is an unordered set of streams; tens of thousands of clients can each hold their own stream and write to the same table concurrently, while readers can read a single stream concurrently at different offsets. A stream comes in one of three types that map directly onto batch-versus-streaming needs: **UNBUFFERED** only acknowledges once data is durably committed and immediately visible; **BUFFERED** acknowledges once data is written but not yet committed, so reads can't see it until it's flushed — suited to small, frequent streaming appends; **PENDING** keeps all written data invisible until the whole stream is explicitly committed, which is how batch ETL gets large-transaction ACID semantics (many workers each write their own PENDING stream, report success to a coordinator, and the coordinator issues one batch commit that makes every stream's data atomically visible together). Clients can optionally supply a `row_offset` when appending, which gives exactly-once semantics if two clients race for the same offset (only one wins) — omitting it trades that guarantee for lower latency by always appending at the stream's end.

Physically, a Stream is an ordered list of **Streamlets** — contiguous row ranges, each replicated across at least two Borg clusters in a region for durability, with only the last Streamlet in a Stream ever writable at one time. Each Streamlet is further split into **Fragments**, contiguous blocks of rows inside a log file stored on Colossus. Vu explicitly maps this onto Kafka's design: a Streamlet plays the role of a replicated partition, and a Fragment plays the role of a same-size Kafka segment file. Data is first written in a **write-optimized storage format (WOS)** and later converted by a dedicated Storage Optimization Service into a **read-optimized storage format (ROS)** — BigQuery tables use the Capacitor format (see [[capacitor-file-format]]) as their ROS, which Vu also likens to Hudi's Avro-then-Parquet two-format pattern.

Vortex splits into a **control plane** and a **data plane**. The control plane is one or more Stream Metadata Servers (SMS), backed by Cloud Spanner, which own the physical metadata of Streams/Streamlets/Fragments plus the table's logical metadata (schema, partition/clustering info); each table is owned by exactly one SMS, and if that SMS becomes unavailable the table is redistributed to a new one. The data plane is the Stream Servers: each owns a set of Streams and their Fragments, keeps in-memory metadata (which Streamlet a Fragment belongs to, its size, min/max record timestamp, whether it's finalized, schema, partition/clustering columns), and persists that metadata via a transaction log plus periodic checkpoints, cleaning up old logs afterward. When a client wants to append, the SMS finds (or creates) a Stream, assigns its Streamlet to a Stream Server chosen to balance CPU/memory/network load across the cluster, and hands the client a long-lived connection to that server. The Stream Server buffers writes to a Fragment up to 2MB before flushing — this prevents many small writes to the file system and helps the Fragment achieve better compression — and if a write fails partway, it finalizes the current fragment, retries on the next one, and ultimately finalizes the whole Streamlet and fails the append, forcing the client back to the SMS for a fresh Streamlet (usually on a different server). Separately, a Fragment's *max size* (a different quantity from the 2MB write buffer) is chosen small enough that WOS→ROS conversion happens frequently, but not so small that the Stream Server has to manage excessive per-fragment metadata.

The **client library** handles retries, decryption, and decompression, and mediates schema evolution: if the table schema changes mid-write, the Stream Server fails the append and tells the client to fetch the new schema from the SMS (the SMS can't push schema changes to clients directly, since clients only talk to Stream Servers — it notifies Stream Servers via heartbeat instead, and they pass the word along on the next append attempt). The library supports both short-lived unary connections (pooled, suited to infrequent appends) and a long-lived bidirectional streaming connection with flow control, where the Stream Server throttles accepted data if Colossus is too slow to keep the buffered-but-uncommitted data from exhausting server memory. Every few seconds, a Stream Server sends the SMS a heartbeat carrying its full Streamlet-metadata snapshot (new log files, size growth) plus its own load (CPU/memory/throughput), which the SMS uses both to reconcile metadata (deleting Streamlet records the Stream Server no longer recognizes) and to balance future Streamlet placement.

The **Storage Optimization Service** treats Fragments as a Log-Structured Merge tree: WOS-format Fragments sit at the bottom, and each conversion pass produces a more ROS-optimized, larger, merged version higher up. Each Fragment carries a `creation_timestamp` and `deletion_timestamp`; a client reading at a given snapshot timestamp only sees Fragments whose interval contains it, and each optimization step atomically closes out the old Fragments' interval while opening the new ones'. Separately, Vortex also offers BigQuery's read-after-write consistency guarantee (the ability to read data immediately after writing it); the source describes both facts without stating that the timestamp-interval mechanism is what produces that guarantee. The same service also re-clusters storage blocks by a table's declared clustering column(s) whenever ordering drifts. **Data verification** pipelines, run as SQL over BigQuery itself, check every Vortex API call end-to-end — that a Stream/Streamlet/Fragment was actually created at the expected offset, and that every record survives its WOS→ROS conversion exactly once with matching input/output.

For **reading**, Dremel talks to Colossus directly via the client library; if a Stream Server is up, it can hand out in-memory metadata telling the client exactly which Fragments and byte ranges to read (never reading past a Fragment's finalized size, to avoid partial writes). Partition pruning works by tracking min/max column-value properties and bloom filters per Fragment/Streamlet on the partitioning/clustering columns — properties the Stream Server maintains live and forwards to the SMS via heartbeat once a Fragment finalizes — and BigQuery evaluates the query's filters against those properties to skip irrelevant Fragments before ever opening a file.

**Data mutation** (UPDATE/DELETE/MERGE) works by marking ranges of rows deleted with a mask rather than rewriting content in place: a DELETE finds candidate rows and persists a deletion mask to the affected Streamlet/Fragment at commit time (sometimes over-marking a few extra rows to keep the mask small — Google calls these over-marked rows "reinserted," and their real copies are committed atomically alongside the mask). UPDATE is DELETE-then-INSERT. Because the Stream Server only reports its status to the SMS asynchronously via heartbeat, there's a window — the **tail** — where the client has already written data the SMS doesn't know about yet; a DML that needs to delete something in that tail causes the SMS to mark the *whole* tail deleted provisionally, then map that down to the real deletion mask once the next heartbeat reports the tail's actual Fragments. A related conflict: the Storage Optimization Service also deletes old Fragments (as part of WOS→ROS conversion or re-clustering) and could race with a DML that has already marked those same rows deleted — Google's fix is that the optimization service simply won't commit while a DML is running, at the cost of blocking WOS→ROS conversion (and thus read performance) if the DML runs long; Vortex mitigates this with a 1:1 Fragment WOS-to-ROS conversion mode.
