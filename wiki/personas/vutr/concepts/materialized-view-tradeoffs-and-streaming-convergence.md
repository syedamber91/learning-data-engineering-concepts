vutr's mental hierarchy runs: a view is just a saved query, re-executed against the base tables on every read; a table stores data physically on disk and is kept current by explicit INSERT/UPDATE/DELETE; a materialized view (MV) is their hybrid — "like their son" — aware of a defining query the way a view is, but persisting the result physically like a table, refreshed whenever the source data it's built from changes. He frames the MV explicitly as a cache: a faster-to-read copy of a slower-to-compute result, inheriting the one problem every cache has — keeping it from going stale — reframed here as keeping the MV synchronized with its base tables.

Every MV decision, in his framing, comes down to exactly two questions: how fresh does the data need to be, and what are you willing to pay for that freshness? The reason those two questions are inseparable is that an MV doesn't make a query cheaper in any absolute sense — it moves the computation from read time to write/refresh time; you're paying for the result in advance so that later reads are fast. Full refresh (recompute the entire view on a schedule, hourly, daily, whatever fits) is the simple, predictable option, but it's stale between runs and pays the whole compute cost on every refresh — expensive at real data scale. Incremental refresh (IVM — see [[incremental-view-maintenance-td-dd-dbsp]]) instead recomputes only what actually changed, which is what makes frequent refreshing affordable in the first place, since the cost per refresh drops in proportion to how small the change is relative to the full dataset — but, vutr stresses, IVM does not make freshness free: refresh cost still scales with how often you refresh, it's just a smaller constant. Support for incremental refresh is also not universal across query shapes even within one engine: he notes BigQuery's MV can only refresh incrementally in specific cases (for example, a JOIN where only the right-hand side gained new rows), falling back to full refresh outside those shapes. He lists ClickHouse, Databricks, and BigQuery as supporting incremental refresh, adding Snowflake with an explicit question mark of his own rather than asserting it confidently. There is, in his view, no single correct point on the freshness-vs-cost dial — a fraud-detection system and an hourly reporting dashboard have genuinely different requirements, and the right answer depends entirely on the use case.

Pushed to its limit — maximum freshness, refreshed as fast as the latency budget allows — the MV concept stops being distinguishable from stream processing, and vutr grounds that convergence in three real systems rather than asserting it in the abstract. Apache Flink's Dynamic Tables model a continuously updating table whose SQL query becomes a "continuous query" — one that never terminates, continually updating its result as new input arrives, itself producing another dynamic table — which he calls semantically equivalent to a materialized view with eager (immediate, on-every-change) view updating. RisingWave, a streaming OLAP database, is built around the identical principle: its materialized views are maintained incrementally by routing the stream of source-table changes through its internal dataflow graph and updating only the rows that change, never re-scanning the whole source. ClickHouse's incremental materialized view is implemented as a trigger: a query that runs on each newly inserted batch of source rows, writes its output into a separate target table, and merges that output with what's already there as more data arrives — a result vutr notes is semantically equivalent to re-running the query over the entire source dataset, just computed piece by piece as the pieces show up, rather than all at once.

His practical conclusion: the next time a use case calls for real-time analytics, the materialized view deserves a place on the same shortlist as a dedicated stream processor like Flink or a specialized real-time database like Pinot — not a separate, lesser category from them — with the actual deciding factor being whether the desired freshness is achievable at a refresh cost you're willing to pay, and the MV's specific appeal being that it lets you stay inside familiar SQL and your existing database rather than adopting a new system.
