Where BigQuery, Snowflake, and Databricks' Photon engine all lean on vectorized batch execution, Redshift takes the other major OLAP execution strategy: code generation. A query arrives at a single coordinator ("leader") node, which parses, rewrites, and optimizes it using the cluster's topology and the underlying data-distribution statistics specifically to avoid unnecessary data movement. The optimized plan is then split into individual execution units, each of which Redshift compiles into query-and-schema-specific C++ — called a segment — and ships as a compiled binary to the worker nodes for execution. Pure per-query code generation would still be wasteful at the data-scanning step, though, since every operator would otherwise need its own generated scan logic — so Redshift adds one precompiled, switch-statement-based, SIMD-vectorized scan layer on top of the generated code, covering all data types once rather than regenerating scan logic per query. That single layer is Redshift's one deliberate concession to vectorization inside an otherwise code-generation-first engine.

Because compiling C++ per query is expensive, Redshift caches compiled segments in the local cluster's cache so repeated or similar queries skip recompilation. In 2020, Redshift added a Compilation Service that moves this cache to a shared, cross-cluster tier backed by separate resources — and the reported effect, per the source paper, was that fleet-wide cache hit rates rose from 99.60% to 99.95%, with 87% of the previously-missed 0.4% actually present in the new external cache rather than needing a fresh compile. AQUA (Advanced Query Accelerator, 2021) pushes this idea one layer further into hardware: it's a multi-tenant caching and acceleration service for Redshift Managed Storage, built on custom Nitro-ASIC servers with FPGAs, that caches hot data locally and executes detected scan/aggregation operations directly against that cached data rather than pulling it from S3 first.

Redshift also automates the maintenance work a database administrator used to do by hand. Automatic Table Optimization (ATO) periodically collects query-execution metadata — optimized plans, cardinalities, predicate selectivities — to recommend a distribution key (minimizing data movement across the workload) and a sort key (reducing bytes read via zone maps), and can apply either recommendation automatically in idle background windows instead of waiting on a manual `ALTER TABLE`. The Automatic Workload Manager (AutoWLM) converts each incoming query's execution plan and statistics into a feature vector, feeds it to an ML model to predict compile time, execution time, and memory need, and uses that prediction to schedule shorter queries ahead of longer ones, admit a query to execution only once its predicted memory need can actually be satisfied, and throttle overall concurrency when utilization gets too high. Priority is enforced with a weighted round-robin scheme where CPU and I/O are split in exponentially decreasing shares across priority levels, and a higher-priority query arriving after a lower-priority one can redeem resources from it mid-flight — though the probability of that preemption shrinks with each occurrence, so a low-priority query is never starved indefinitely. The Query Predictor Framework is the piece that actually trains and serves those ML models (using XGBoost) from data collected inside the cluster, which the source notes also feeds the Compilation Service's own optimization. A Query Rewriting Framework (QRF) does double duty: it rewrites the execution order of unions/joins/aggregations for a submitted query, and it's the same machinery behind Redshift's automatic materialized-view maintenance, where Redshift decides when to refresh an MV by weighing the view's utility in the observed workload against its refresh cost, maintains filter/select/group-by/join changes incrementally rather than by full recompute, and auto-rewrites incoming queries to transparently hit the MV instead of the base tables. Underneath all of this sits a "smart" warm pool of pre-provisioned EC2 instances — one pool per Availability Zone per region — whose size Redshift forecasts with an ML model, so that failed-node replacement, cluster resumption, and Concurrency Scaling all draw from an already-warmed pool instead of paying cold-start latency; vutr notes this is conceptually close to Snowflake's own pre-warmed worker pool, just implemented with a learned forecasting model rather than a fixed policy.

Across everything he researched, vutr is explicit that Redshift is, to his knowledge, the only OLAP system that openly documents using machine learning for its own background operations (AutoWLM's and ATO's forecasting) — he notes neither Snowflake nor BigQuery mention doing the same, though he flags that as an observation about what's publicly documented, not a claim that the others don't do anything similar internally.
