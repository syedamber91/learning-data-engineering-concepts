Start with the misconception: adding a broker to a Kafka cluster does not balance it. The new broker sits idle and underutilized until someone creates new topics or partitions on it — or manually moves existing partitions onto it. That "manually" is the whole problem. Because Kafka tightly couples compute and storage, a partition's replicas live on specific brokers' local disks, so any membership change (broker added, broker died) or any hot-spot fix means physically copying partition data across the network via [[leader-follower-replication]].

**The native mechanism.** Kafka ships `bin/kafka-reassign-partitions.sh`, which runs in three modes: `--generate` takes a list of topics and brokers and produces a candidate reassignment plan; `--execute` applies a plan (the generated one, or a hand-written one); `--verify` checks the status of every partition listed in the last execute. The user drives every step. It is error-prone, plans only from the input you give it, and — as PayPal found — the default behavior repartitions *all* partitions, including ones on perfectly healthy brokers. At PayPal's scale (85+ clusters, 1,500 brokers, 20,000+ topics, ~2,000 MirrorMakers, traffic peaking around 1.3 trillion messages daily during the 2022 Retail Friday), that default could make a cluster unavailable for *days*. Their fix: modify reassignment to move only the under-replicated partitions on affected brokers. They applied the same idea to patching — a plugin checks for under-replicated partitions before a restart, so clusters patch in parallel but only one broker per cluster at a time.

**Cruise Control automates the planning, not the physics.** At LinkedIn, operating ~7K+ brokers made manual balancing untenable, so Cruise Control periodically gathers broker- and partition-level resource utilization, builds a workload model of the cluster, and lets a goal optimizer explore proposals against a user-defined goal list — far more robust than the native tool's input-only planning. But the data movement issue is still there. Replicating partition data is slow, and the tool only acts on a *snapshot* of the cluster; by the time a decision executes, the cluster state it was based on may have changed significantly, making the decision less accurate. That is why Cruise Control's goals are tuned to be strict and effective mainly when traffic fluctuates little — under significant load variation it struggles.

**AutoMQ's answer: make reassignment a metadata edit.** With brokers made stateless via [[automq-wal-shared-storage]] (data on object storage, one replica per partition), reassignment moves no data — the controller just rewrites the broker↔partition mapping in KRaft-style metadata (see [[zookeeper-to-kraft-metadata-management]]). Decisions execute quickly and accurately because nothing waits on replication. On top of this, AutoBalancer self-balances continuously. Each goal has a threshold and an acceptable range — e.g. CPU utilization threshold 50% ±20% means 30–70% counts as balanced — split into detection goals (capacity violations on CPU or network I/O) and optimization goals (producer, consumer, and QPS balance), and prioritized as hard (must always hold, like partition-count or traffic caps) or soft (droppable on conflict, like traffic balance). Candidate actions are scored mathematically and the highest-scoring one runs. Three components implement it: a Metrics Collector that ships broker metrics to the controller over an internal topic; a State Manager on the controller maintaining a ClusterModel from KRaft metadata plus those metrics; and a Decision Scheduler that wakes on an interval (e.g. every 60 seconds), snapshots the ClusterModel, finds goal-violating brokers, and executes feasible reassignment plans. New brokers are warmed up gradually rather than flooded, placement is rack-aware, and slow brokers get load shed to healthier nodes.

**LinkedIn's answer: shrink the unit.** Northguard, Kafka's successor at LinkedIn (150 clusters, 10,000 machines, 400,000 topics, 17PB/day), replicates segments instead of partitions — see [[northguard-segment-level-replication]]. Since segments are created far more frequently than logs, adding a broker requires no movement at all: the system simply assigns the next *active* segment to the new broker and ingest traffic follows. The trade-off framing is consistent across all three sources: you either automate the plan (Cruise Control), delete the data movement (AutoMQ), or make the moved unit small enough not to hurt (Northguard).
