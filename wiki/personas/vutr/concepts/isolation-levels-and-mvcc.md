Isolation is the one ACID property that gets genuinely harder as concurrency rises, and the notes lay out a ladder of three levels — read committed, snapshot isolation, and serializability — each one closing off anomalies the level below it still allows.

Read committed is the floor: a transaction only ever sees data that has actually been committed, which rules out dirty reads and dirty writes. The mechanism is unglamorous — row-level locks. A transaction that wants to write a row must acquire that row's lock, and any other transaction wanting to modify the same row has to wait. Reads are handled more cheaply: rather than block a reader behind a long-running writer, the database keeps two versions of the row around — the last committed version and the new value the writer currently holds the lock on — and serves the old version to concurrent readers. That two-version trick is a preview of the next level.

Read committed still allows a non-repeatable read: read the same row twice inside one transaction and get two different values, because some other transaction committed a change in between — even though both reads only ever saw committed data. Snapshot isolation (SI) closes that gap by giving every transaction a consistent snapshot: it sees only changes committed before the transaction started; everything from concurrent or later transactions is invisible, no matter when it eventually commits. Implementing that snapshot means keeping more than the two versions read committed needed — a whole lineage of row versions, one for whichever point in time some open transaction might still need to read from. That's exactly why SI is described as multi-version concurrency control (MVCC): the "multi-version" storage *is* the mechanism the snapshot guarantee is built on, not a separate feature layered on top of it.

That version-keeping is also why SI has a lopsided relationship with locking: it needs write locks but not read locks. A reader never has to wait for a writer, because it never looks at the writer's in-progress version at all — it just reads whichever older, already-committed version was current when its own transaction began. A writer, though, must still take a lock before changing a row, or two concurrent writers could each stomp on the other's change — a dirty write, or the subtler lost update, where T1 and T2 both read a balance of $1,000, T1 commits +$200 to make it $1,200, and T2 — still working from its original $1,000 read — commits +$300 to make it $1,300, silently erasing T1's update. Locking the row on write is what stops that; nothing has to lock the row on read, because reads are served from a snapshot instead of from live, in-progress state.

SI still has a hole: write skew, where two transactions each read a true precondition, each act correctly on what they read, and each commits — yet the combination violates an invariant neither transaction could see, because they read one object and wrote another. The canonical case is a meeting-room booking system: two clients each check "is Room A free from 1–2pm," both see yes, both insert a booking for that slot, and Room A ends up double-booked — because reading "is it available" never locked the room against a concurrent insert. Only serializability, the top of the ladder, closes that gap, guaranteeing that whatever order transactions actually run in, the *effect* is as if they had run one at a time. Two-phase locking (2PL) gets there pessimistically: shared locks for reads (many readers can hold one together), exclusive locks for writes (only one writer, and it blocks readers too), all held until the transaction ends — correct, but writes now stall reads as well as other writes. Serialized snapshot isolation (SSI) gets there optimistically instead: transactions run against an SI-style snapshot exactly as before, with no locks taken up front, and only at commit time does the database check whether anything the transaction read was concurrently changed by someone else; if so, it aborts the transaction and forces a retry rather than let write skew slip through.

PostgreSQL is the concrete case that makes all of this physical rather than abstract. To keep the "previous version" that SI/MVCC needs, PostgreSQL doesn't overwrite a row in place on UPDATE — it leaves the existing row version where it is, marks it no longer the live version, and inserts an entirely new row version carrying the updated data; DELETE works the same way, marking the row "deleted" rather than physically removing it. In both cases the old bytes are still sitting on disk, because some other transaction's snapshot might still need to read exactly that version. Nothing reclaims that space at commit time — that's a separate, asynchronous job: PostgreSQL's VACUUM process, which garbage-collects old row versions once no live transaction's snapshot can possibly need them anymore. Skip vacuuming and the table doesn't become logically wrong, it becomes physically bloated: every UPDATE and DELETE is, underneath, an insert that leaves its predecessor behind for MVCC's sake, and something has to eventually clean that trail up.
