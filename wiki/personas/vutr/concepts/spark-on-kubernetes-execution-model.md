Vu's starting point for running Spark on [[kubernetes|Kubernetes]] is to notice there are really two different "clusters" involved, and conflating them is where confusion starts. The **Spark cluster** is the application-level pairing of a **driver** — the JVM process that manages the whole Spark application, handling user code, planning execution, and distributing tasks — and its **executors**, the processes that actually run the tasks the driver hands them and report status/results back. That Spark cluster has to run somewhere physically, and providing that "somewhere" is the job of a separate **resource cluster** managed by a cluster manager; Kubernetes is one option among several a Spark application can target.

Submitting a Spark application to Kubernetes routes the request to the k8s **API server**, carrying the same configuration a user would pass to any `spark-submit` (e.g., driver/executor resources). In **cluster** deploy mode, Kubernetes launches a driver Pod inside the cluster itself, and that driver Pod then asks Kubernetes — now acting as the cluster manager — to launch executor Pods per the submitted configuration. In **client** mode, the driver instead stays on the client machine, which pushes a real networking obligation onto the user: making sure executor Pods spun up inside the cluster can actually connect back to the driver process running outside it. Once submitted, the rest proceeds as an ordinary Spark job — the driver plans and schedules the application's logic, and physical execution is handed to the executors. After the job finishes, Kubernetes garbage-collects the executor Pods, but the driver Pod is deliberately left in a persisted "Completed" state until someone manually cleans it up — specifically so its logs remain available for debugging after the fact.

Vu's own hands-on path used [minikube](https://minikube.sigs.k8s.io/docs/) to stand up a local cluster rather than a real (expensive) one, which also installs and configures `kubectl` automatically. Getting a Spark application actually running required four ingredients: a **Spark image** (built and pushed via the `docker-image-tool.sh` script bundled with the downloaded Spark package, since anything running on k8s must be containerized); the **k8s API endpoint** to submit against, obtained with `kubectl cluster-info`; **RBAC** setup, since minikube runs with RBAC enabled by default — a dedicated service account bound to a role (Vu used the built-in `edit` clusterrole for the exercise, while explicitly flagging that this is not necessarily the right role for a real deployment, where the principle of least privilege should govern the choice) so Kubernetes will actually let the submission create driver and executor Pods; and an application to run (a bundled example jar). The `spark-submit` command ties all of this together via `--master k8s://<endpoint>`, `--deploy-mode cluster`, `--conf spark.kubernetes.container.image=<image>`, and `--conf spark.kubernetes.authenticate.driver.serviceAccountName=<account>` — the same configuration keys carry over unchanged into a PySpark `SparkSession` when submitting programmatically instead of via the CLI.

Historically, Spark applications couldn't be expressed as ordinary Kubernetes YAML manifests the way a Deployment or StatefulSet can (see [[kubernetes-workload-abstractions]]), because a Spark job's driver/executor pair has a transient, dynamic lifecycle fundamentally unlike a long-running microservice. The **Spark Operator** closed that gap by extending the Kubernetes API itself: an Operator works by defining a **Custom Resource Definition (CRD)** — in this case a new resource type called `SparkApplication` — plus a **custom controller** that watches for changes to resources of that new type and reconciles the cluster's actual state to match, the same underlying control-loop idea that drives every built-in Kubernetes controller. With the Spark Operator installed, a Spark application can be declared entirely in a YAML manifest (`kind: SparkApplication`) submitted like any other Kubernetes resource, rather than invoked through `spark-submit` directly.
