The vault's existing shuffle material explains *why* map output gets written to disk and *why* it needs a decoupling service ([[shuffle-writes-to-disk-and-external-shuffle-service]]), but flags the actual write mechanics as an open gap. Vu's dedicated shuffle article fills that in directly, using MapReduce's own "mapper"/"reducer" naming (mappers output shuffle data, reducers read it) even though, as he's careful to note, MapReduce didn't invent shuffling — the pattern predates it in DBMSs.

Before Spark 1.2, the hash shuffle manager handled this the naive way: each map task hashes the value of the shuffle key (e.g. `user_id`) per record to decide which downstream reduce partition it belongs to, then writes that record to a separate file dedicated to that one reduce task. With M map tasks and R reduce tasks — R being `spark.sql.shuffle.partitions`, 200 by default regardless of actual data size — this produces M×R files. At real scale (1,000 map tasks × 1,000 reduce tasks = 1,000,000 intermediate files) the sheer I/O volume of opening, writing, and closing that many small files becomes the bottleneck itself, independent of the data volume being shuffled.

Since Spark 1.2.0, the sort shuffle manager is the default, and it collapses that file explosion: each mapper writes exactly one data file, with records sorted and grouped by destination partition/reducer ID, plus a companion index file recording each partition's start/stop byte offsets inside that single file. This is faster specifically because it turns the workload into sequential disk I/O (one file, contiguous partition ranges) instead of hash shuffle's scattered random writes across many small files. The trade isn't unconditional, though: when the number of reducers is small, hashing is still cheaper than sorting and the file count stays manageable, so Spark falls back to the old hash-style behavior automatically whenever the partition count sits below `spark.shuffle.sort.bypassMergeThreshold` (default 200) — writing separate per-reducer files and merging them into one combined file afterward, index file included. Project Tungsten later added a third variant on top of the sort manager: same data-file-plus-index-file output shape, but operating directly on serialized binary data without ever deserializing it into JVM objects, which is faster still — though only under specific conditions the source doesn't enumerate beyond linking to the implementation.

Once a mapper's shuffle file is written, it registers that file's location with the Driver — this is the same registration mechanism [[shuffle-writes-to-disk-and-external-shuffle-service|the external shuffle service]] later intercepts to decouple serving from executor lifetime. On the read side, the Driver hands each reducer the map-output locations it needs (e.g. Reduce Task #7 must pull every mapper's "Partition #7" data). If a needed file lives on a remote executor, the reducer issues a TCP request to that mapper host's shuffle service; the service reads the index file to find the requested partition's byte offset inside the data file and returns just that slice. The reducer pulls the result into a memory buffer and builds an iterator over it for downstream processing. The real cost here is structural, not incidental: a single reduce task typically has to make this same round trip against *every* mapper that produced data for its partition, which is both network overhead on the reducer side and repeated disk-read load on every mapper host it touches — the same many-to-one fan-in problem [[uber-remote-shuffle-service|Uber's Remote Shuffle Service]] was built to eliminate by making mappers write to one shared remote destination instead.

Two further tuning levers sit downstream of the mechanism itself, both about shrinking or right-sizing what actually crosses the network. First, reducing the data before it shuffles at all: `reduceByKey()` pre-aggregates within each partition before anything crosses the network, while `groupByKey()` ships every raw value across the network with no pre-aggregation — both are wide transformations requiring a shuffle, but `reduceByKey()`'s associative/commutative pre-combine step means substantially less data actually gets shuffled. Second, sizing the reducer count itself: with no official Spark-doc figure for target partition size, Vu adopts an external suggestion of 100–200MB per partition, computing `partitions = data size / target partition size` and then sanity-checking that figure against Spark's own guidance of 2-3 tasks per CPU core (so a 150-partition target implies roughly 50-75 cores, not more). Two lower-level buffer knobs round this out, both trading memory pressure for fewer I/O round trips: `spark.shuffle.file.buffer` (default 32KB) is the mapper's in-memory buffer before it flushes to its shuffle output file — bigger means less frequent disk writes but more per-mapper memory held; `spark.reducer.maxSizeInFlight` is the reducer's fetch buffer for pulled mapper output — bigger means fewer, larger network requests per reduce task but more memory reserved to receive them.
